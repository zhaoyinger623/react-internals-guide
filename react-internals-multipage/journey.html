一次 setState 的完整旅程

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
这一章先说明 setState 之后依次发生哪些事情。这里暂不展开源码细节；后续章节会按照 Update、Lane、Render 和 Commit 等环节分别讲解。

从点击到屏幕,一共 10 步

👆 你点了按钮

浏览器触发事件。React 把事件统一委托到根容器,反查到是哪个组件、要调哪个 onClick。 深入:第五层·合成事件 →

↓
📞 调用 setState

你的 onClick 里执行 setState。React 据「什么事件触发的」给这次更新分配一个优先级 Lane(点击=最高优)。 深入:Schedule·Lane →

↓
📥 产生 Update,入队

生成一个 Update 对象挂到该组件 Fiber 的 UpdateQueue,并把 Lane 一路冒泡到 root,标记「这个应用有活了」。 深入:Trigger·UpdateQueue →

↓
⏱ Scheduler 安排

调度器决定「什么时候做、先做谁」。它把渲染工作切成一片约 5ms,片与片之间把控制权还给浏览器绘制/响应——所以页面不卡。 深入:Schedule·Scheduler →

↓
🡓 Render·beginWork

从上往下遍历 Fiber 树:重新执行你的组件拿到新 JSX,和旧的做 Diff,生成/复用子 Fiber。这一步只算,不碰真实 DOM。 深入:Render·Fiber 架构 →

↓
🡑 Render·completeWork

走到底再往上归:为原生节点创建离屏 DOM、把子 DOM 拼上,并记录之后需要执行的 DOM 变更。 深入:Render·变更标记 →

↓
🌳 得到一棵带标记的新树

Render 结束,内存里有了一棵记录了变更的新 Fiber 树(workInProgress)。此时屏幕还没变——这棵树可以被中断、被丢弃重来,用户看不见。 深入:Render·双缓存 →

↓
✅ Commit 落地

按照这些变更标记一次性、同步地改真实 DOM(插入/更新/删除),并把 current 指针切到新树。屏幕在这一刻变化。 深入:Commit·提交阶段 →

↓
🎬 effect 执行

useLayoutEffect 在绘制前同步跑(避免闪烁)→ 浏览器绘制 → useEffect 在绘制后异步跑。 深入:第一层·生命周期 →

↓
🖥 屏幕更新

用户看到了最新界面。一次 setState 的旅程结束。

章节对应关系：UpdateQueue 对应第 3 步，Scheduler 对应第 4 步，Fiber、Diff 和 Hooks 主要发生在第 5 步，Commit 对应第 8 步。

读后续章节前，先分清三件事

① Render 与 Commit

Render：执行组件、比较新旧结果，算出哪些节点需要新增、更新或删除。此时不修改页面。

Commit：把已经算好的结果应用到 DOM，页面在这个阶段发生变化。

重点：Render 负责“算”，Commit 负责“改”。

② 为什么只有 Render 能中断

Render 操作的是内存中的 workInProgress Fiber 树。暂停、重新计算或丢弃它，都不会让用户看到不完整的页面。

Commit 直接修改 DOM。如果执行到一半暂停，页面可能同时包含一部分旧结果和一部分新结果。

因此：Render 可以中断，Commit 必须连续完成。

③ Lane 与 Scheduler

Lane：标记更新的优先级，决定本次 Render 要处理哪些更新。

Scheduler：安排任务何时运行，并判断当前任务是否需要让出主线程。

重点：Lane 选择“做哪些更新”，Scheduler 安排“什么时候做”。
