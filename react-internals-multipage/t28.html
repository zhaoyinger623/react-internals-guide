28 · 性能诊断与工程化

🗺 本章路线图

本节内容：使用 Profiler 和 Web 性能指标定位渲染与交互问题。

本章包括：DevTools Profiler、Web 指标(LCP/INP/CLS)、内存泄漏与 effect 清理、Suspense 数据获取。

用 React DevTools Profiler 定位问题

  • Profiler 面板:录制一次交互,看每个组件的渲染次数、耗时、为什么渲染(props/state/hooks 变化)。开「Highlight updates」直观看到哪些组件在无谓重渲染。
  • <Profiler> API:代码里包住子树,onRender 回调拿到 actualDuration 等指标,做自动化性能监控。
<Profiler id="list" onRender={(id, phase, actualDuration) => {
  log(id, phase, actualDuration);   // phase: mount | update
}}>
  <List/>
</Profiler>

核心 Web 指标与 React 的关系

指标含义React 抓手
LCP最大内容绘制SSR/流式、代码分割、图片优化
INP交互到下一次绘制并发(useTransition)、避免长任务、时间切片
CLS布局偏移占位尺寸、避免 hydration 后跳动

常见内存泄漏与 effect 清理

  • 订阅/定时器/事件监听未在 effect 返回函数里清理 → 组件卸载后仍持有引用。
  • 异步请求返回时组件已卸载仍 setState → 用 AbortController 或忽略标志。
  • 闭包捕获旧值(stale closure)→ 依赖数组写全、或用 ref/函数式更新。

Suspense 做数据获取

配合支持 Suspense 的数据层(RSC、React Query、Relay 等),组件用 use() 或库的 hook「读」数据,未就绪自动挂起显示 fallback,把 loading 状态从组件内 if(loading) 上移到声明式的 <Suspense> 边界,配合并发做「避免 loading 闪烁」。

这一层的知识关系：内部更新机制是基础；SSR、RSC 和并发涉及运行位置与执行时机；Profiler 和 Web 指标用于验证性能判断；最后再根据场景选择 Context 或外部 store、CSR 或 SSR/RSC、手动记忆化或 Compiler。
本章小结：用 Profiler 看谁在无谓重渲染;INP 靠并发/切片优化;effect 没清理订阅/定时器会泄漏;Suspense 把 loading 从组件内上移到声明式边界。→ 补充专题:Context 源码。
