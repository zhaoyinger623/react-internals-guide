3 · 生命周期完整执行顺序

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

生命周期不是一组需要背诵的 API，而是组件从计算到提交、再到清理的执行顺序。

① Render 与 Commit 的边界 · ② 类组件挂载/更新/卸载 · ③ Effect 的创建与清理 · ④ useEffect 与 useLayoutEffect 的时机 · ⑤ StrictMode 为什么会多执行一次。

先看一次更新的时间顺序

1 · Render执行组件，计算新 JSX 和 Fiber。可以暂停或重新开始，不能修改 DOM。
2 · Mutation把 Render 算好的结果真正写入 DOM，例如新增、修改或删除节点。
3 · LayoutDOM 已更新、浏览器尚未绘制。执行 layout effect 和类组件 DidMount/DidUpdate。
4 · Passive浏览器获得绘制机会后，React 再执行普通的 useEffect。
“组件渲染完成”可能指不同时间点：组件函数执行完，只代表 Render 完成；DOM 修改完成发生在 Commit；用户看到画面还要等浏览器绘制；useEffect 通常在此后执行。

类组件：挂载、更新与卸载

场景Render 阶段Commit 阶段常见用途
挂载constructor → getDerivedStateFromProps → render修改 DOM → componentDidMount初始化 state；挂载后建立订阅或读取 DOM。
更新getDerivedStateFromProps → shouldComponentUpdate → rendergetSnapshotBeforeUpdate → 修改 DOM → componentDidUpdate决定是否更新；在 DOM 改变前保存滚动位置；提交后同步外部系统。
卸载—componentWillUnmount → 删除 DOM取消订阅、定时器和未完成任务。

为什么三个旧生命周期被标记为 UNSAFE

旧方法都在 Render 阶段执行。同步渲染时代，Render 通常会紧接着 Commit，于是开发者很容易在这些方法里建立订阅、发请求或读取 DOM。但现代 React 可以暂停或放弃一次 Render，调用了 will 方法，并不代表这次更新一定会提交。

例子：在 componentWillMount 中建立订阅 
componentWillMount() {
  chat.subscribe(this.props.roomId); // 有外部副作用
}
① 开始渲染 room=A调用 willMount，建立订阅 A。
② 渲染被暂停或放弃例如子组件 Suspense；A 对应的界面从未提交。
③ React 重新渲染再次调用 willMount，又建立一个订阅。
④ 产生泄漏第一次没有真正挂载，因此不能依赖正常卸载流程撤销那次操作。

问题不在“调用了两次”，而在“副作用发生了，但对应的 Commit 可能根本不存在”。把订阅放入 componentDidMount 后，只有节点真正提交到界面时才会建立订阅，并可由 componentWillUnmount 对称清理。

旧方法常见的不安全写法为什么有问题替代位置
componentWillMount发请求、订阅、操作全局对象方法执行后，组件不一定真的挂载。初始化 state 放 constructor；副作用放 componentDidMount。
componentWillReceiveProps收到新 props 就请求数据或同步 state这组 props 对应的 Render 可能被更新的 props 取代，永远不提交。提交后的同步放 componentDidUpdate；少量派生 state 使用 getDerivedStateFromProps，通常优先在 render 中直接计算。
componentWillUpdate启动动画、操作 DOM、重建订阅它可能针对一次最终未提交的更新提前产生副作用。DOM 改变前读取信息用 getSnapshotBeforeUpdate；提交后的操作放 componentDidUpdate。
准确说法：这些 API 不是因为“旧”而失效，也不是突然全部删除；React 将它们重命名为 UNSAFE_componentWillMount、UNSAFE_componentWillReceiveProps 和 UNSAFE_componentWillUpdate，用于兼容历史代码。UNSAFE 表示它们不适合承载依赖“这次更新一定会提交”的逻辑，新代码不应使用。

函数组件：Effect 的完整生命周期

useEffect(() => {
  // setup：连接外部系统
  const connection = connect(roomId);

  return () => {
    // cleanup：撤销上一次 setup 做的事情
    connection.disconnect();
  };
}, [roomId]);
发生什么执行顺序读取到的值
首次挂载提交界面 → 执行 setup首次渲染的 roomId
roomId 改变先执行上一次 cleanup → 再执行新的 setup旧 cleanup 读取旧值；新 setup 读取新值
依赖没有变化setup 和 cleanup 都不执行—
组件卸载执行最后一次 cleanup最后一次 setup 对应的值
依赖数组控制同步频率：不传依赖数组时，每次提交后执行；传 [] 时只在挂载后 setup、卸载时 cleanup；传 [a,b] 时，任一依赖通过 Object.is 比较发生变化才重新同步。

为什么 Effect 不能只理解为“挂载后执行”

假设聊天室组件当前连接 general 房间，用户随后把 roomId 改成 travel。组件没有卸载，但旧连接必须断开，并建立新连接：

① roomId = generalEffect setup：连接 general。
② 用户选择 travel组件重新渲染，界面显示 travel。
③ 先执行旧 cleanup断开 general，避免同时保留两个连接。
④ 再执行新 setup连接 travel，使外部连接与当前 props 一致。

因此，一个 Effect 更适合看成一对操作：setup 建立某种外部连接，cleanup 撤销这次连接。依赖变化时，这一对操作会重复发生；它不只对应挂载和卸载。

需要 Effect

useEffect(() => {
  const c = connect(roomId);
  return () => c.disconnect();
}, [roomId]);

聊天室连接是 React 之外的系统，需要在 props 改变后重新同步。

不需要 Effect

// 直接由本次 props 计算
const total = price * quantity;

total 只是已有数据的计算结果，不需要先渲染一次，再通过 Effect 写回 state。

StrictMode 下为什么会多执行一次

React 额外执行一次，是为了让两类原本偶发的 bug 在开发时立即出现。正确的组件应该满足：

  • 组件函数执行两次，结果仍相同。组件不能在 Render 中修改 props、全局变量或其他外部数据。
  • Effect 可以安全地 setup → cleanup → setup。cleanup 必须完整撤销上一次 setup。

检查 1：Render 是否纯净

function List({ items }) {
  items.push('新项目'); // ❌ 修改了 props 数组
  return items.map(...);
}

普通执行一次时，错误可能不明显。StrictMode 连续执行两次后，“新项目”会重复出现，直接暴露 Render 修改了外部数据。

检查 2：Effect 是否正确清理

useEffect(() => {
  socket.connect();
  // ❌ 没有 return disconnect
}, []);

StrictMode 执行 setup → cleanup → setup。因为 cleanup 缺失，最终会保留两个连接，问题立刻可见。补上 return () => socket.disconnect() 后，任意时刻只保留一个连接。

为什么模拟“建立 → 清理 → 再建立”？真实应用中，用户可能进入页面、离开页面、再回来；依赖变化时 Effect 也会反复清理和建立。StrictMode 提前做一次压力测试，确认代码能承受这个过程。这项检查仅在开发环境执行，生产环境不会因此多渲染一次页面。
本章小结：Render 只计算；Commit 修改 DOM。Layout Effect 在绘制前同步运行，普通 Effect 在提交后异步运行。依赖变化时先清理旧 Effect，再建立新 Effect；卸载时执行最后一次清理。下一章进入React 更新的四个阶段。
