阶段三 · Render / 12

12 · flags：记录 Render 的变更

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图(先建立脉络,再看细节)

「副作用(effect)」= 组件在纯计算之外要对外界做的事:改真实 DOM、跑 useEffect、更新 ref。

核心结论：Render 阶段在 Fiber 上设置 flags，记录之后需要执行的操作；Commit 阶段读取这些 flags，并修改 DOM 或执行相关副作用。

为什么分两步:Render 可中断、可丢弃重来,不能真去动 DOM;所以先把「要做的事」攒成标记,等 Commit 一次性做完。

本章回答三个问题:① 有哪些标记? ② 标记记在哪、Commit 怎么高效找到它们? ③ 什么时候执行?(第 ③ 步的执行细节在下一章 Commit·提交阶段)

先回答：一个“有变化的 Fiber”怎么产生

不是拿 baseState 直接做 Diff。一次组件更新在 Render 阶段内部按下面顺序发生：

  1. 先算 state：processUpdateQueue 从旧 state 出发处理本轮 update，得到新的 memoizedState。baseState 只在有 update 被跳过时作为未来重算起点。
  2. 再执行组件：组件使用新 state/props 返回新的 React Element，例如从 <h1>0</h1> 变成 <h1>1</h1>。
  3. 再做子节点 Diff：reconcileChildren 比较旧子 Fiber 与新 React Element，决定复用、新建、移动或删除哪些 WIP Fiber。
  4. 再比较 DOM 属性：completeWork 比较宿主 Fiber 的旧 props 与新 props，发现文本、className、style 等差异。
  5. 最后留下 flags：新增/移动产生 Placement，删除产生 Deletion，属性/文本变化产生 Update；effect deps 变化产生 Passive 或 Layout 相关标记。
因此答案是：state 由 UpdateQueue 计算；新 JSX 由组件执行产生；Fiber 结构变化由 Diff 发现；DOM 属性变化由 completeWork 发现。它们不是同一步，但都发生在 Render 阶段，并最终把操作记录到 WIP Fiber。

① flags 是 Render 留给 Commit 的操作清单

Render 发现具体变化后，在对应 WIP Fiber 的 flags 中记录“Commit 应该执行什么”：

标记(flag)含义Commit 何时处理
Placement新增 / 移动节点(insertBefore/appendChild)Mutation
Update属性/文本变更(updateProperties)Mutation
Deletion删除节点(18 存在 父.deletions 数组里)Mutation
Passive有 useEffect 要执行Commit 后异步
Layout(LayoutMask)有 useLayoutEffect / DidMount/UpdateLayout
Ref需要更新 refLayout
effect 中再次 setState 会怎样？执行 useEffect/useLayoutEffect 本身不会自动产生 update；只有 effect 回调内部主动调用 setState 才会。useLayoutEffect 的回调在 Commit 的 Layout 阶段同步执行；useEffect 的回调在 Passive 刷新时执行。setState 会创建一个全新的 update、分配 Lane、进入队列并重新安排 root，随后开启下一轮 Render。React 不会在当前 Diff 中途直接修改 state。
本轮 Render → 本轮 Commit
                 ↓ effect 回调调用 setState
新 update 入队 → Lane → Schedule → 下一轮 Render → 下一轮 Commit

防止无限循环：如果 effect 每次执行都无条件 setState，下一轮 Render 又让该 effect 再次执行，就会不断产生更新。依赖数组和条件判断的作用之一，就是让这个闭环在状态同步完成后停止。

② 标记记在哪、怎么高效找到:位运算 + subtreeFlags

先搞清楚「打在哪」:每个 fiber 就是一个对象,它身上有个字段叫 flags(一个数字)。Render 阶段 React 每发现一个 fiber 有变化,就在它自己的 flags 上记一笔——「我要被插入」记 Placement、「我属性变了」记 Update。标记是每个 fiber 各自带的,不是全局一张表。

例子：h1 文本变化，ul 中新增一个 li。先看叶子自己的 flags，再从下往上汇总 subtreeFlags：

HostRoot
subtreeFlags = Update | Placement
↑ App 汇报：我的子树中有 Update 和 Placement
App
flags = NoFlags
subtreeFlags = Update | Placement
↑ h1 汇报 Update
h1（文本变化）
flags = Update
subtreeFlags = NoFlags
↑ ul 汇报 Placement
ul
flags = NoFlags
subtreeFlags = Placement
↑ li 汇报 Placement
li（新增）
flags = Placement
subtreeFlags = NoFlags

读取方法：flags 只写“我自己要做什么”；subtreeFlags 只写“我的后代中有什么操作”。completeWork 从叶子向父级返回时完成汇总，Commit 据此跳过没有变更的分支。

flags 是位掩码(bitmask):一个整数就能同时携带多个标记,用位运算叠加和检查,比数组/字符串快得多:

// 每个标记占二进制的一位
Placement = 0b0000010;  Update = 0b0000100;  Passive = 0b0001000; …
fiber.flags |= Placement;       // 叠加一个标记
fiber.flags |= Update;          // 再叠加一个(现在同时带两个)
if(fiber.flags & Update){ … }   // 检查是否带某标记

subtreeFlags 不是另一份操作，它只是父节点的“目录”

假设 App 有两个子分支：Header 没变化，List 中有一个新增 li。

Header.flags = NoFlags
Header.subtreeFlags = NoFlags

li.flags = Placement
List.subtreeFlags = Placement
App.subtreeFlags = Placement

completeWork 从子节点返回父节点时，执行近似计算：

parent.subtreeFlags |= child.flags;
parent.subtreeFlags |= child.subtreeFlags;

Commit 从 App 往下检查时：

  1. 看到 Header.subtreeFlags === NoFlags 且 Header 自己也没有相关 flag，直接跳过整个 Header 分支。
  2. 看到 List.subtreeFlags 含 Placement，进入 List。
  3. 最终找到 li.flags = Placement，执行插入 DOM。
关键区别：li.flags 是“请插入我”；List.subtreeFlags 是“我的后代里有人需要处理”。Commit 最终执行的是具体 Fiber 自己的 flags，subtreeFlags 只用于导航和剪枝。
