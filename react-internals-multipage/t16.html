16 · 三大模块完整协作

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

Scheduler(WHEN)→ Reconciler(WHAT)→ Renderer(HOW)三层解耦,任务像流水线一样流转。

本章包括：三层各自职责、解耦价值、任务流转全链路(附源码调用链图)。

Scheduler / Reconciler / Renderer 三层解耦,任务像流水线一样流转:

Scheduler 调度层

纯 JS 时间/优先级管理,不认识 React。产出:在时间片里调用 callback。

→
Reconciler 协调层

平台无关。beginWork/completeWork + Diff,产出带 flags 的 WIP 树。

→
Renderer 渲染层

平台相关。react-dom / native / 三方,据 flags 落地。

📊 源码调用链全景(ReactFiberWorkLoop)

这张图把从「输入(触发更新)」到「输出(react-dom)」的真实函数调用链画出来了 —— 全文的骨架都在这里。

① 触发 → ② 调度(注册到 Scheduler)→ ③ Scheduler 回调里驱动 render(sync / concurrent 两条入口 → performUnitOfWork 循环构建 fiber 树)→ ④ commitRoot 输出到 react-dom

解耦的价值

  • Scheduler 不知道 Fiber,只调度"一个会返回续作的函数",可单独用于任何切片场景。
  • Reconciler 只输出"要做什么"(抽象增删改),不碰具体 API,通过 HostConfig 适配层调用 Renderer。
  • 换 Renderer 就能换平台:同一套核心渲染 DOM / 原生 / Canvas / 3D。
// 任务流转全链路(bind 把它们串起来)
scheduleCallback(prio, performConcurrentWorkOnRoot.bind(null, root))
   Scheduler 在时间片里 → performConcurrentWorkOnRoot
      → renderRootConcurrent → workLoopConcurrent → beginWork/completeWork  [Reconciler]
      → commitRoot → commitMutationEffects → HostConfig.appendChild/...      [Renderer]
本章小结：Scheduler 只调度「会返回续作的函数」、Reconciler 通过 HostConfig 调 Renderer、换 Renderer 就换平台;调用链 scheduleCallback→performConcurrentWorkOnRoot→workLoop→commitRoot。→ 下一章:第五层·批量更新。
