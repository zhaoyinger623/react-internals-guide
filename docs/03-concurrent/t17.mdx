## 三种批量更新

时期

行为

React 16/17 同步批量

只在 React 事件回调内批处理(靠 `batchedUpdates` + executionContext 标志)

React 16/17 异步"漏网"

`setTimeout`/`Promise`/原生事件里的多次 setState **不批处理**,各触发一次渲染

React 18 自动批量

## ① React 16/17:靠「执行上下文标志」批处理(所以会漏网)

16/17 里,React 给**自己的事件回调**套了一层 `batchedUpdates`:进回调前把全局标志 `executionContext` 打上「批处理中」,回调里的 setState 就**只入队、不刷新**;回调结束再统一 flush 一次。

```js
function batchedUpdates(fn){
  const prev = executionContext;
  executionContext |= BatchedContext;   // 打上「批处理中」标志
  try { return fn(); }             // 你的事件回调:多次 setState 只入队
  finally {
    executionContext = prev;            // 回调结束,清标志
    if(executionContext === NoContext) flushSyncCallbacks(); // 统一刷一次
  }
}
```

**为什么 setTimeout/Promise 里会「漏网」:**这些回调是**浏览器/微任务队列直接调**的,**不经过** React 的 `batchedUpdates` 包裹 → `executionContext` 没打标志 → 每个 setState 都当场各刷一次渲染。

## ② React 18:靠「微任务 + 去重」批处理(所以哪都能批)

18 不再依赖执行上下文标志。每次 setState → `scheduleUpdateOnFiber` → `ensureRootIsScheduled`。对 SyncLane,它用 **微任务**安排 flush,而且会**去重**:

```js
function ensureRootIsScheduled(root){
  const newPriority = getHighestPriorityLane(getNextLanes(root));
  if(root.callbackPriority === newPriority) return; // ★同优先级已排过 → 直接返回(去重!)
  if(newPriority === SyncLane) scheduleMicrotask(flushSyncCallbacks); // 安排一个微任务
  else scheduleCallback(…);
  root.callbackPriority = newPriority;
}
```

因为 flush 挂在**「微任务 / 本轮事件循环末尾」**,而不是「React 回调末尾」,所以 setTimeout、Promise、原生事件里的多次 setState 也落在同一批、只刷一次——这就是「**任何地方都自动批处理**」。

## ③ 一步步看:Promise 里连写 3 次 setState

Promise.resolve().then(() => &#123; setA(); setB(); setC(); &#125;)

setState #1 → 标 SyncLane → ensureRootIsScheduled:没排过安排微任务 · 计1次

setState #2 → 标 SyncLane → 已排过·同优先级return 去重 · 仍1次

setState #3 → 同上return 去重 · 仍1次

—— 本轮同步代码跑完 ——

微任务执行 → flushSyncCallbacks → performSyncWorkOnRoot一次 render,3 个 update 一起 reduce

**结果:只渲染 1 次 ✨**

## ④ flushSync:主动跳出批处理

```js
flushSync(() => setCount(1));  // 立即同步渲染,不等微任务
// 下一行就能读到更新后的 DOM(如需立刻测量布局)
console.log(ref.current.offsetHeight);
```

flushSync 内部临时设好执行上下文,并在结束时**立刻** `flushSyncCallbacks`,不等微任务。代价是丢掉批处理的性能收益,只在确实需要「同步拿到结果」时用。

## ⑤ 并发(Transition)下的批量

`startTransition` 里的多次 setState 落在同一个 **TransitionLane**,同样只渲染一次。但如果一批里**混了不同优先级**(比如一个 Sync、一个 Transition),会**分成两批各自渲染**——高优先渲并上屏,低优随后。这也是靠 Lane 区分的。

### 互动:一个回调里 setState ×3 → 一次渲染

**怎么操作：**点击一次按钮，分别观察“进入队列的 update 数量”和“真正发生的 Render 次数”。本例要说明三次状态更新不等于三次渲染。

**本章小结：**React 18 靠 **Lane + 微任务去重**实现「任何地方都自动批处理」;同一事件循环里的多次 setState 落同一 lane、只安排一次渲染;`flushSync` 可跳出。→ 下一章:[第五层·Suspense](/03-concurrent/t18)。
