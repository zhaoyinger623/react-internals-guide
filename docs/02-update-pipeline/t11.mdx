Commit 把 Render 阶段打好的 flags **一次性、同步**落到屏幕,分 BeforeMutation → Mutation → Layout 三段,**不可中断**。

本章包括：**三段各做什么、current 指针何时切、useEffect 何时异步跑**。

Render 算完得到带 flags 的 WIP 树,Commit **同步、一气呵成**地执行,分三段:

**Render 时只是登记，之后才真正运行：**组件执行到 `useEffect(() => &#123;...&#125;, deps)` 或 `useLayoutEffect(() => &#123;...&#125;, deps)` 时，React 先把箭头函数和 deps 保存到 Fiber，并记录对应标记。**箭头函数里的代码此时还没有运行。**DOM 提交后，React 在 Layout 阶段运行 `useLayoutEffect` 保存的函数；普通 `useEffect` 保存的函数则在后续 Passive 阶段运行。

**1\. BeforeMutation**

修改 DOM 前读取快照，例如类组件 `getSnapshotBeforeUpdate`

→

**2\. Mutation**

真正改 DOM:增删改。**屏幕在此刻变化**。此段末切换 `current` 指针

→

**3\. Layout**

DOM 已更新。同步执行 `useLayoutEffect` 回调、`DidMount/Update` 和 ref

**Commit 三段结束后：**普通 `useEffect` 对应的 Passive effect 由 `flushPassiveEffects` 在后续任务中执行。通常浏览器会先绘制；某些由用户交互触发的更新中，React 也可能在绘制前刷新 effect。关键区别是：它不属于上面三段中的同步 DOM 提交流程。

## useEffect：Render 登记，提交后执行

Render 执行组件时，`useEffect` 会比较 deps、保存 effect 对象，并在需要执行时给 Fiber 添加 `Passive` 标记。Commit 看到该标记后确保后续会调用 `flushPassiveEffects`；真正的 effect 回调通常在同步 Commit 结束后异步执行：先清理上一次 effect，再执行本次 effect。

`Render：useEffect(create, deps)` → 保存 create/deps、打 Passive 标记
`Commit：提交 DOM` → 收集待执行的 Passive effect
`Commit 之后：flushPassiveEffects()` → 执行 cleanup 和 create

所以“Hooks 在 Render 阶段”与“effect 在 Commit 附近执行”并不冲突：前者指 Hook API 被组件调用并建立内部记录，后者指其中保存的副作用函数何时真正运行。

Commit 不可中断,是因为它直接操作真实 DOM 并触发生命周期 —— 中途停下会让用户看到**撕裂的、不一致的中间画面**,还可能重复触发副作用。

**一次更新完成：**Commit 依次处理 DOM 变更前的快照、Mutation 和 Layout，并切换 current；普通 `useEffect` 在提交后调度。下一章再以并发渲染解释这四个阶段如何被高优先级更新影响。
