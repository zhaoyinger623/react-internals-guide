# Lane：选择本次更新

> 本章建立 Lane 的完整工作模型：为什么需要 Lane → Lane 如何分配给 update → 如何传播到 root → `getNextLanes` 如何选择一批工作 → Render 如何筛选 update 和子树 → 挂起、唤醒、过期与提交后 Lane 如何变化。

## 1. React 为什么需要 Lane

同一个 root 上可以同时存在不同来源、不同紧急程度的更新。例如用户正在输入文字，同时页面还要重新筛选一个很大的列表：

| 更新 | 用户感受 | 合理处理方式 |
| --- | --- | --- |
| 输入框显示新字符 | 用户正在等待反馈 | 尽快完成 |
| 大列表重新筛选 | 计算量大 | 可以放入 `startTransition`，稍后或分片处理 |

如果 React 只有一个普通 FIFO 队列，就会陷入两个问题：

1. 严格按顺序执行，前面的大列表可能阻塞后面的输入更新。
2. 直接把低优更新挪到队尾，又可能破坏同一 state 队列原本的计算顺序。

Lane 的解决方式不是移动 update，而是给每个 update 附加一个二进制优先级位。Render 遍历队列时，只执行本轮选中的 Lane；没选中的 update 仍按原顺序保存在 `baseQueue`，以后重放。

> 核心关系：UpdateQueue 保存“更新发生的顺序”；Lane 决定“本轮选择哪些更新”；baseQueue 保存“被跳过、以后必须按原顺序重放的更新”。

## 2. Lane、Lanes 与 renderLanes

| 名称 | 含义 | 常见位置 |
| --- | --- | --- |
| `Lane` | 一个二进制位，通常表示一条 update 所属的优先级通道 | `update.lane` |
| `Lanes` | 多个 Lane 按位 OR 得到的集合，不是一种新的优先级 | `fiber.lanes`、`root.pendingLanes` |
| `renderLanes` | 当前 Render 选择处理的 Lane 集合，可能包含一个或多个 bit | Render 工作循环、UpdateQueue |

Lane 不是线程、state 或 Scheduler 任务。它本质上是数字中的一个 bit，React 用位运算完成集合的加入、相交、移除和优先级查找。

```js
SyncLane       = 0b0001
DefaultLane    = 0b0100
TransitionLane = 0b1000

pendingLanes = SyncLane | TransitionLane
// 0b0001 | 0b1000 = 0b1001
```

`0b1001` 的意思是“Sync 和 Transition 都还存在”，不是产生了一个名叫 `1001` 的新优先级。

## 3. 创建 update 时如何分配 Lane

调用 `setState` 后，React 会创建 update，并通过 `requestUpdateLane()` 根据当前执行上下文决定 Lane。

| 更新来源 | 常见 Lane | 调度含义 |
| --- | --- | --- |
| 点击、键盘输入等离散交互 | `SyncLane` | 用户正在等待结果，通常尽快处理 |
| 拖动、滚动等连续交互 | `InputContinuousLane` | 优先级较高，但与离散同步更新区分 |
| 普通更新 | `DefaultLane` | 常规优先级 |
| `startTransition` 内的更新 | 某个 `TransitionLane` | 允许等待、暂停或被更紧急更新抢占 |
| Suspense 恢复后的重试 | 某个 `RetryLane` | 再次尝试之前挂起的边界 |

完整起点是：

```text
事件或 Transition 上下文
  → requestUpdateLane()
  → update.lane
  → update 进入 UpdateQueue
```

Lane 不改变 update 在队列中的位置，只决定它何时有资格被某次 Render 处理。

## 4. Lane 如何从 Fiber 传播到 root

假设组件树为 `Root → App → Main → Button`，Button 产生一个 Sync update：

```text
Button.lanes       |= Sync   // Button 自己有 Sync 工作
Main.childLanes    |= Sync   // Main 的后代有 Sync 工作
App.childLanes     |= Sync   // App 的后代有 Sync 工作
root.pendingLanes  |= Sync   // 整个 root 有 Sync 工作未完成
```

这几个字段承担不同职责：

- `fiber.lanes`：这个 Fiber 自己有哪些更新。
- `fiber.childLanes`：它的后代中有哪些更新。
- `root.pendingLanes`：整个 root 上还有哪些 Lane 没完成。

`childLanes` 还是 Render 的导航索引。如果某个分支的 `lanes` 和 `childLanes` 都不包含本轮 Lane，React 可以 bailout，直接跳过整棵子树。

## 5. root 如何记录 Lane 的运行状态

`pendingLanes` 只表示“还没完成”。为了判断一项工作现在能不能执行，root 还维护其他集合：

| root 字段 | 含义 | 典型变化时机 |
| --- | --- | --- |
| `pendingLanes` | 已到达 root、尚未全部完成 | update 到来时加入，成功完成后移除 |
| `suspendedLanes` | 本次尝试因数据等原因挂起 | Render 遇到 Suspense 后标记 |
| `pingedLanes` | 原本挂起，但依赖已经就绪，可以重试 | Promise resolve 后标记 |
| `expiredLanes` | 等待过久，需要防止继续饿死 | 到达过期时间后标记 |
| `entangledLanes` | 与其他 Lane 有共同完成关系 | 相关更新不应独立提交中间状态时建立关联 |

可以把它们理解成同一批工作在不同状态下的索引，而不是五条独立的 update 队列。

## 6. getNextLanes 如何选择下一批工作

`getNextLanes(root)` 不是简单执行“取最高的一个 bit”。它要选出下一批值得执行、现在能够执行、并且适合一起执行的工作。

简化后的判断顺序如下：

1. `pendingLanes === NoLanes`：没有工作，直接返回。
2. 先考虑非 Idle 工作，避免普通页面更新没完成就去做后台空闲任务。
3. 优先从尚未挂起的 Lane 中选择。
4. 如果没有新鲜可执行工作，再检查已经 ping、可以重试的 Lane。
5. 从候选集合中识别最高优先级组；返回值可能包含多个 Lane。
6. 如果已有 WIP，只有新工作优先级更高时才值得丢掉现有进度并切换。
7. 准备 Render 时，再计算需要共同完成的 entangled Lane。

> “选择最高优先级”不等于“只选一个 Lane”。`getHighestPriorityLane` 返回一个 bit，用来识别最高优先级；`getHighestPriorityLanes` 和后续 entanglement 逻辑可以得到一组 Lane。

### 哪些 Lane 会一起处理

| 情况 | 结果 | 原因 |
| --- | --- | --- |
| 同一优先级组中的多个 Lane | 可以作为一批选择 | 减少重复遍历并协调同组工作 |
| 高优和普通低优 Lane 同时等待 | 通常先处理高优组 | 避免低优大任务拖慢输入反馈 |
| Lane 彼此 entangled | 有效 Render 范围纳入关联 Lane | 避免提交不应出现的中间状态 |
| 特殊强制同步刷新 | 可以把指定额外 Lane 一起刷新 | 满足历史导航、滚动恢复等宿主场景 |

以常见的输入加列表筛选为例：

```js
root.pendingLanes = SyncLane | TransitionLane;
renderLanes = SyncLane;
```

这不是因为一次 Render 只能处理一个 Lane，而是调度策略。假设输入更新只需 2ms，大列表需要 80ms；总是合并会让用户等待整个大列表。React 通常先完成 Sync，再继续 Transition。

在 entanglement 或特殊同步刷新场景中，同一批有效 Render 范围也可以包含 Sync 和 Transition。是否提交还取决于整次 Render 是否完成、是否挂起。

## 7. Render 如何用 Lanes 处理 update

处理 UpdateQueue 时，React检查 `update.lane` 是否属于本轮有效 Lanes：

```js
if (isSubsetOfLanes(renderLanes, update.lane)) {
  state = applyUpdate(state, update); // 本轮执行
} else {
  cloneToBaseQueue(update);           // 本轮跳过，留待重放
}
```

因此，一次处理多个 Lane 不需要另一套队列算法。集合里包含几个 Lane，就能匹配对应的几类 update。

如果某个低优 update 被跳过，后面的 update 即使本轮执行，也可能需要克隆进 `baseQueue`。这是为了以后从 `baseState` 开始恢复原始顺序。具体重放规则见上一章“UpdateQueue：更新如何入队”。

Lane 还参与子树 bailout：若 Fiber 自己和后代都没有与本轮 Lanes 相交的工作，React 可以复用旧子树，不执行该分支的组件和 Diff。

## 8. 一组 Lane 从产生到结束

| 时刻 | Lane 状态 |
| --- | --- |
| update 创建 | 写入 `update.lane` |
| 传播到 root | 合并进 `fiber.lanes`、`childLanes`、`root.pendingLanes` |
| 开始调度 | `getNextLanes` 选择下一批工作 |
| Render | 匹配的 update 执行；不匹配的进入 `baseQueue` |
| Render 挂起 | 相应 Lane 进入 `suspendedLanes` |
| 数据就绪 | Lane 进入 `pingedLanes`，获得重试机会 |
| 等待过久 | Lane 可能进入 `expiredLanes`，防止饿死 |
| 成功提交 | 已完成且没有剩余工作的 Lane 从 `pendingLanes` 等集合移除 |

Lane 负责描述“哪些工作属于同一批”和“哪批更紧急”。真正安排回调时间、检查 `shouldYield()`、暂停或继续 Fiber 工作循环的是 Scheduler 与 Reconciler 的协作。

## 9. 用位运算观察 Lane 集合

<LanePlayground />

常见位运算对应四类问题：

```js
pendingLanes |= lane;                 // OR：加入集合
(renderLanes & updateLane) !== 0;     // AND：是否相交
remainingLanes = lanes & ~finished;   // AND NOT：移除已完成集合
highestLane = lanes & -lanes;         // 取最右侧的 1，识别最高优先级 bit
```

最后一行只得到一个最高 bit，不代表 `getNextLanes` 的最终结果一定只有一个 Lane。React 会根据这个 bit 所属的优先级组、挂起状态、正在进行的 WIP 和 entanglement 得到真正的下一批工作。

## 10. 对照源码时看哪些位置

- `ReactFiberLane.js`：Lane 常量、`getNextLanes`、挂起/唤醒/过期/完成状态。
- `ReactFiberWorkLoop.js`：准备 Render、比较 WIP Lanes、计算 entangled Render 范围。
- `ReactFiberConcurrentUpdates.js`：更新 Lane 向 root 传播。
- `ReactFiberClassUpdateQueue.js` / `ReactFiberHooks.js`：用 render Lanes 执行或跳过 update。
- `ReactFiberBeginWork.js`：通过 `lanes` 与 `childLanes` bailout 子树。

> 本章结束后应形成的模型：update 带一个 Lane；Fiber 和 root 汇总成 Lanes；root 每次选择一批有效 Lanes；Render 只处理这批更新并跳过无关子树；未处理的更新仍按原顺序保留，直到后续 Render 完成。
