这里的“渲染”是 React 调用组件并取得返回的 JSX。它属于 Render 阶段，只计算下一版界面；真正修改 DOM 要等到 Commit。

<SectionTitle>函数组件与类组件如何完成渲染</SectionTitle>

| 对比 | 类组件 | 函数组件 |
| --- | --- | --- |
| React 如何调用 | 创建实例后调用 `instance.render()` | 直接把函数作为普通函数调用 `Component(props)` |
| 状态保存位置 | 实例的 `this.state`，更新保存在实例 Fiber 的 UpdateQueue | Fiber 的 `memoizedState` 指向 Hook 链表 |
| 每次 Render | 复用同一个实例并再次调用 `render` | 函数从头执行，局部变量和函数重新创建 |
| 为什么状态不丢 | 实例与对应 Fiber 被保留 | Hook 状态保存在 Fiber 上，而不是函数局部变量中 |

### 类组件：状态在实例上

```js
class Counter extends Component {
  state = { count: 0 };
  render() {
    return <button>{this.state.count}</button>;
  }
}
```

同一个 `Counter` 实例会跨更新复用，`this.state` 因而持续存在。

  </ContentCard>
  <ContentCard tone="success">

### 函数组件：状态在 Fiber 上

```js
function Counter() {
  const [count] = useState(0);
  return <button>{count}</button>;
}
```

函数每次都重新执行，但 React 按顺序从 Fiber 的 Hook 链表中取回 `count`。

  </ContentCard>
</div>

<Callout>

函数组件不是因为闭包把状态永久保存下来。闭包保存的是**某一次 Render 的变量快照**；真正跨 Render 保存状态的是 Fiber 与 Hook 数据结构。

</Callout>

<SectionTitle tone="green">函数整体重新执行意味着什么</SectionTitle>

```js
function Product({ price, quantity }) {
  const total = price * quantity;
  const handleBuy = () => submit(total);
  return <button onClick={handleBuy}>购买 {total}</button>;
}
```

每次 Render 都会重新计算 `total`、创建新的 `handleBuy`，并返回新的 React Element。React 随后比较新旧描述，决定子 Fiber 如何复用。

这并不意味着真实 DOM 一定重建。**组件函数重新执行、Fiber 重新协调、DOM 真正变化是三个不同层次。**若比较后属性没有变化，Commit 可以不执行 DOM 更新。

### stale closure 为什么会出现

```js
function Counter() {
  const [count, setCount] = useState(0);

  function showLater() {
    setTimeout(() => alert(count), 1000);
  }
}
```

`showLater` 捕获的是创建它那一次 Render 的 `count`。即使一秒内又发生了更新，旧函数仍读取旧快照。这是 JavaScript 闭包语义，不是 React 随机返回旧值。

| 需要什么 | 推荐方式 | 原因 |
| --- | --- | --- |
| 根据最新 state 更新 | `setCount(c => c + 1)` | 更新函数由 React 在处理队列时读取最新基础状态 |
| 在异步回调中读取最新值 | 用 `ref.current` 保存最新值 | ref 对象跨 Render 保持同一引用 |
| 保持传给 memo 子组件的函数稳定 | `useCallback` | 避免每次创建新引用破坏 props 浅比较 |
| 缓存昂贵计算结果 | `useMemo` | 依赖不变时复用计算结果，而不是跳过组件本身 |

<Callout type="warning">

`useCallback` 和 `useMemo` 不是为了阻止函数组件重新执行。它们只在组件执行过程中复用某个函数引用或计算结果；是否跳过整个组件，要看 `memo` 和 Fiber bailout 条件。

</Callout>

<SectionTitle tone="purple">什么会触发重新渲染</SectionTitle>

| 来源 | 发生什么 | 是否能被 `memo` 阻止 |
| --- | --- | --- |
| 组件自身 state 更新 | Update 进入本组件的 Hook/Class 队列 | 不能 |
| 父组件传入的新 props | 父组件执行后，子组件收到新的输入 | props 浅比较相等时可以 |
| 读取的 Context 改变 | Context 依赖被标记，消费者需要重新计算 | 不能仅靠 props 相等阻止 |
| 父组件重新渲染 | 默认继续执行它的子组件 | 使用 `memo` 且 props 相等时可以 |

<SectionTitle tone="orange">bailout 如何跳过无效工作</SectionTitle>

`memo` 比较新旧 props；若 props 相等、组件自身没有匹配本轮 Lane 的更新、Context 依赖也没变化，React 可以复用当前 Fiber 的结果并跳过组件函数。

```js
const Child = memo(function Child({ user, onSelect }) {
  return <button onClick={onSelect}>{user.name}</button>;
});
```

如果父组件每次都创建新的 `user` 对象或 `onSelect` 函数，浅比较仍会认为 props 改变。此时才需要根据真实性能瓶颈考虑 `useMemo` 或 `useCallback`，而不是无差别地缓存所有值。
