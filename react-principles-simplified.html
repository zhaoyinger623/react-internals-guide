读后续章节前，先分清三件事

① Render 与 Commit

Render：执行组件、比较新旧结果，算出哪些节点需要新增、更新或删除。此时不修改页面。

Commit：把已经算好的结果应用到 DOM，页面在这个阶段发生变化。

重点：Render 负责“算”，Commit 负责“改”。

② 为什么只有 Render 能中断

Render 操作的是内存中的 workInProgress Fiber 树。暂停、重新计算或丢弃它，都不会让用户看到不完整的页面。

Commit 直接修改 DOM。如果执行到一半暂停，页面可能同时包含一部分旧结果和一部分新结果。

因此：Render 可以中断，Commit 必须连续完成。

③ Lane 与 Scheduler

Lane：标记更新的优先级，决定本次 Render 要处理哪些更新。

Scheduler：安排任务何时运行，并判断当前任务是否需要让出主线程。

重点：Lane 选择“做哪些更新”，Scheduler 安排“什么时候做”。

const root = createRoot(document.getElementById('root'));  // ① 建两个根对象

启动过程分为三步

  1. 创建两个根对象 —— createRoot(#root) 创建应用根 fiberRootNode 和 Fiber 树根 hostRootFiber，并建立双向引用。
  2. 把 <App/> 变成一次「更新」 —— root.render(<App/>) 生成一个 Update,挂到 hostRootFiber 的更新队列上,再喊一声 scheduleUpdateOnFiber(“有活了,安排渲染”)。

与双缓存的关系：首次渲染时，hostRootFiber 是 current 树的起点，React 在内存中构建 workInProgress 树。提交完成后，fiberRoot.current 指向新树。详见Render·双缓存。
本节过程：创建应用根和 Fiber 树根 → 将 <App/> 放入根节点的更新队列 → 触发首次渲染（mount）。

1 · 组件渲染机制

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

这里的“渲染”指 React 调用组件并取得组件返回的 JSX。本章介绍以下三部分：

① 函数 vs 类组件怎么被渲染、状态存哪 · ② 什么情况会重新渲染(state/props/context/父级)· ③ 怎么阻止无效重渲染(memo / useMemo / useCallback + bailout)。

函数组件与类组件如何完成渲染

「渲染」= React 调用你的组件,拿到返回的 JSX(React 元素)。区别只在怎么被调用、状态挂在哪:

类组件函数组件
如何渲染实例化后调用 instance.render()直接把函数当普通函数调用 Component(props)
状态存哪实例 this.state + updateQueueFiber 的 memoizedState 上的 hooks 链表
每次渲染同一实例,方法引用不变函数整体重新执行一遍,内部变量/函数全部重建

🗄️「状态存哪」图解:state 到底放在哪个对象上

「存储」= 你的 count 这类状态,React 把它保存在内存里的哪个对象上,下次渲染再从那里读回来。两种组件放的地方不同:

👉 要点：类组件把状态放在「组件实例 this」上;函数组件没有实例,状态被 React 存到「这个组件对应的 Fiber 节点」的 memoizedState 链表里。所以函数每次重跑,还能从 Fiber 上把上次的 count 读回来。

函数组件更新时会再次执行整个函数。理解这一点后，才能继续分析 Hooks 的取值方式、闭包中的旧值以及 useCallback 的作用。

①「函数整体重新执行」到底发生了什么

每次状态变化,React 会再调用一次你的函数组件。函数体里除了 Hook 从「链表」里读到最新值外,所有局部变量、所有内联函数都是全新创建的——和上一次渲染没有任何关系:

function Counter(){
  const [count, setCount] = useState(0);   // count 从 hooks 链表读到「本次」的值

  // ↓ 下面这些,每次渲染都重新算/重新 new 一个:
  const doubled = count * 2;              // 全新的局部变量
  const handleClick = () => setCount(count + 1); // 全新的函数对象(引用每次都不同)

  return <button onClick={handleClick}>{count} · {doubled}</button>;
}

handleClick 捕获的是创建该函数时那次渲染中的 count。第一次渲染创建的函数读取 count=0，第二次渲染创建的新函数读取 count=1。这属于 JavaScript 的闭包行为：每次执行组件函数都会形成一组对应本次渲染的变量和函数。

② 闭包陷阱(stale closure):最常见的 bug 来源

问题出在:某个「旧渲染」里创建的函数,后来还在被使用,它读到的却是那一次的旧值。经典例子:

function Timer(){
  const [count, setCount] = useState(0);
  useEffect(() => {
    const id = setInterval(() => {
      setCount(count + 1);   // ❌ count 永远是 0 → 永远设成 1
    }, 1000);
    return () => clearInterval(id);
  }, []);                     // 空依赖:effect 只在「第一次渲染」跑一次
}

因为依赖数组是 [],这个 setInterval 的回调是第一次渲染时创建的,它闭包里的 count 被永远定格在 0。之后即使界面上 count 变了,这个旧回调用的还是 0,于是计数器卡在 1。两种修法:

// 修法A:函数式更新,不依赖闭包里的 count
setCount(c => c + 1);   // c 是 React 给的「最新值」,与闭包无关 ✅
// 修法B:把 count 放进依赖数组,让 effect 随 count 重建(拿到新闭包)
}, [count]);

③ 为什么需要 useCallback / useMemo

既然每次渲染 handleClick 都是新函数、新引用,那把它传给一个 React.memo 包裹的子组件时,子组件做 props 浅比较会发现「函数引用变了」→ 误以为 props 变了 → 白白重渲染。useCallback 就是让这个函数跨渲染保持同一个引用(依赖不变时返回上一次那个),从而让 memo 生效:

// 不用 useCallback:每次渲染都是新函数 → <Child memo> 每次都重渲染
const handleClick = () => setCount(c => c + 1);
// 用 useCallback:deps 不变时,引用稳定不变 → Child 被 memo 跳过
const handleClick = useCallback(() => setCount(c => c + 1), []);

useMemo 同理,只是缓存的是「计算结果的值/对象」而非函数(详见Render·Hooks)。

④ 对比:类组件为什么没这些问题

类组件的实例只创建一次并一直复用。this、方法引用在多次 render 之间保持同一个;this.state 是一个可变对象,你读 this.state.count 总能拿到最新值。所以类组件里:

  • 方法引用天然稳定,不会因为「重新执行函数体」而每次都变(但要注意手动 bind / 箭头属性)。
  • 没有「函数组件那种每帧一套闭包」的问题——代价是换成了 this 指向、生命周期分散等另一类心智负担。

重新渲染的全部触发条件

  • state 变化:setState / useState 的 dispatch(且新旧值不同,详见 bailout)。
  • props 变化:父组件重渲染,把新 props 传下来。注意:父重渲染,子默认也会重渲染,哪怕 props 没变。
  • context 变化:Provider 的 value 变了,所有消费该 context 的组件重渲染(绕过 memo)。
  • 父组件重渲染:这是最容易被忽略的一条——默认情况下父渲染 = 整棵子树重渲染。
  • 强制:类组件 forceUpdate、hooks 里 useReducer dispatch 相同引用也可能触发一次。

什么是 bailout(提前退出)? React 处理每个组件时会先问一句:「它有必要重渲染吗?」如果 props 引用没变、自身没有待处理的 state 更新、context 也没变,React 就直接复用上一次的结果、跳过它连同整棵子树——这个「跳过」动作就叫 bailout。React.memo 命中时就是靠它跳过。完整条件见第六层·渲染优化。

2 · 虚拟 DOM 的真实含义

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

虚拟 DOM 不是为了「快」,是为了分层解耦、跨平台、可中断。本章包括：

① 它到底为了什么 · ② 和真实 DOM 的关系(JSX → React Element → Fiber → 真实 DOM)· ③ React vs Vue 两种更新哲学的取舍。

最大误解:虚拟 DOM 不是为了"快"。手写精准的原生 DOM 操作永远比"生成 VDOM → Diff → 再操作 DOM"更快。虚拟 DOM 的真正价值是架构。

它到底为了什么

  • 分层架构 / 解耦:用一个 JS 对象描述 UI,把"UI 长什么样"和"怎么落地"分开。上层只关心声明式的树。
  • 跨平台:同一棵 VDOM/Fiber 树,可以交给 react-dom 渲染成 DOM、react-native 渲染成原生视图、react-three-fiber 渲染成 3D。核心不碰平台。
  • 可中断渲染:VDOM/Fiber 是纯 JS 内存对象,可以在内存里慢慢构建、随时暂停丢弃重来,而不触碰真实 DOM。这是 Concurrent 的地基。

虚拟 DOM 与真实 DOM 的关系

从你写的 JSX 到屏幕上的真实 DOM,中间要经过三次「变身」,每一步产物不同、职责不同:

  1. JSX → React Element:JSX 只是语法糖。Babel 把 <div className="x">hi</div> 编译成 React.createElement('div',{className:'x'},'hi'),执行后得到一个轻量、不可变的普通 JS 对象——这就是 React Element(「虚拟 DOM」的最小单位)。它只描述「长什么样」,不含任何真实节点。
  2. React Element → Fiber 节点:Reconciler 拿这个描述去创建/更新对应的 Fiber 节点。Fiber 更「重」,带 child/sibling/return 指针和工作信息,是可中断遍历、承载 state 的载体。
  3. Fiber → 真实 DOM：首次挂载时，Renderer 创建 DOM 节点并写入初始属性，Fiber 的 stateNode 指向该节点；Commit 阶段再把节点插入页面。更新时，Render 阶段比较新旧 props，Commit 阶段应用差异。
三者的区别：React Element 描述期望的界面；Fiber 保存组件状态、树结构和当前工作进度；真实 DOM 是浏览器中已经提交的界面。渲染先产生新的 Element，React 将其与旧 Fiber 对比，再决定需要执行哪些 DOM 操作。

深入对比:React vs Vue 的两种更新哲学

这是中立的架构取舍,不是谁更好。两者都用虚拟 DOM,但更新的思路截然不同——理解这点能帮你看清各自的性能特征和心智模型。

核心分歧:「谁变了我不知道,全量重算」vs「我精确知道谁变了」

React · 运行时全量 Diff

状态一变,React 不知道具体哪里变了,于是把整个组件函数重新执行一遍生成新 VDOM,再和旧 VDOM 做 Diff 找差异。靠运行时对比决定更新什么。为了让「全量重算」不卡顿,才需要 Fiber + 时间切片把大 Diff 摊到多帧。

Vue · 响应式依赖追踪

Vue 的响应式系统在渲染时记录了每个组件/effect 依赖了哪些数据。某个数据变了,Vue 精确知道只有依赖它的地方需要更新,定向触发,不必从父到子全量重算。

Vue 3 的编译期优化:让 Diff「作弊」

Vue 用模板(template)而非 JSX,模板结构在编译期就能被静态分析,于是能做 React 的 JSX 难做的优化:

优化做什么
静态提升(hoistStatic)把不会变的节点提到渲染函数外,只创建一次、每次渲染复用,不参与 Diff
PatchFlag(补丁标记)编译期给动态节点打标记(如「只有 class 会变」),运行时只 diff 标记的那部分属性,跳过静态内容
Block Tree(区块树)把动态节点收集进一个扁平数组,更新时只遍历动态节点,跳过整片静态子树,把 Diff 从「树规模」降到「动态节点规模」
事件缓存(cacheHandlers)内联事件回调被缓存,避免每次渲染都当成「新 prop」

那 React 拿什么对标?

JSX 可以包含任意 JavaScript，编译器能够获得的静态结构信息通常少于模板。过去 React 项目常使用 React.memo、useMemo 和 useCallback 手动减少重复计算；React Compiler（第七层·React Compiler）则在编译阶段分析组件并自动加入记忆化。两种框架的优化路径不同，不宜只用“谁更快”概括。

所以「Vue 的 VDOM 不如 React 纯粹」是什么意思? 指 React 把一切交给运行时的 VDOM Diff,不依赖模板编译期的靶向更新——VDOM 是它唯一且纯粹的更新路径;Vue 则把 VDOM 和编译期优化混用,让 VDOM「掺了水」,不那么「纯 VDOM」。这是褒是贬取决于视角:纯粹 = 心智简单、可中断、跨平台一致;不纯粹 = 运行时更省、开箱即用更快。两条路都很成功。

3 · 生命周期完整执行顺序

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

生命周期不是一组需要背诵的 API，而是组件从计算到提交、再到清理的执行顺序。

① Render 与 Commit 的边界 · ② 类组件挂载/更新/卸载 · ③ Effect 的创建与清理 · ④ useEffect 与 useLayoutEffect 的时机 · ⑤ StrictMode 为什么会多执行一次。

先看一次更新的时间顺序

1 · Render执行组件，计算新 JSX 和 Fiber。可以暂停或重新开始，不能修改 DOM。
2 · Mutation把 Render 算好的结果真正写入 DOM，例如新增、修改或删除节点。
3 · LayoutDOM 已更新、浏览器尚未绘制。执行 layout effect 和类组件 DidMount/DidUpdate。
4 · Passive浏览器获得绘制机会后，React 再执行普通的 useEffect。
“组件渲染完成”可能指不同时间点：组件函数执行完，只代表 Render 完成；DOM 修改完成发生在 Commit；用户看到画面还要等浏览器绘制；useEffect 通常在此后执行。

类组件：挂载、更新与卸载

场景Render 阶段Commit 阶段常见用途
挂载constructor → getDerivedStateFromProps → render修改 DOM → componentDidMount初始化 state；挂载后建立订阅或读取 DOM。
更新getDerivedStateFromProps → shouldComponentUpdate → rendergetSnapshotBeforeUpdate → 修改 DOM → componentDidUpdate决定是否更新；在 DOM 改变前保存滚动位置；提交后同步外部系统。
卸载—componentWillUnmount → 删除 DOM取消订阅、定时器和未完成任务。

为什么三个旧生命周期被标记为 UNSAFE

旧方法都在 Render 阶段执行。同步渲染时代，Render 通常会紧接着 Commit，于是开发者很容易在这些方法里建立订阅、发请求或读取 DOM。但现代 React 可以暂停或放弃一次 Render，调用了 will 方法，并不代表这次更新一定会提交。

例子：在 componentWillMount 中建立订阅 
componentWillMount() {
  chat.subscribe(this.props.roomId); // 有外部副作用
}
① 开始渲染 room=A调用 willMount，建立订阅 A。
② 渲染被暂停或放弃例如子组件 Suspense；A 对应的界面从未提交。
③ React 重新渲染再次调用 willMount，又建立一个订阅。
④ 产生泄漏第一次没有真正挂载，因此不能依赖正常卸载流程撤销那次操作。

问题不在“调用了两次”，而在“副作用发生了，但对应的 Commit 可能根本不存在”。把订阅放入 componentDidMount 后，只有节点真正提交到界面时才会建立订阅，并可由 componentWillUnmount 对称清理。

旧方法常见的不安全写法为什么有问题替代位置
componentWillMount发请求、订阅、操作全局对象方法执行后，组件不一定真的挂载。初始化 state 放 constructor；副作用放 componentDidMount。
componentWillReceiveProps收到新 props 就请求数据或同步 state这组 props 对应的 Render 可能被更新的 props 取代，永远不提交。提交后的同步放 componentDidUpdate；少量派生 state 使用 getDerivedStateFromProps，通常优先在 render 中直接计算。
componentWillUpdate启动动画、操作 DOM、重建订阅它可能针对一次最终未提交的更新提前产生副作用。DOM 改变前读取信息用 getSnapshotBeforeUpdate；提交后的操作放 componentDidUpdate。
准确说法：这些 API 不是因为“旧”而失效，也不是突然全部删除；React 将它们重命名为 UNSAFE_componentWillMount、UNSAFE_componentWillReceiveProps 和 UNSAFE_componentWillUpdate，用于兼容历史代码。UNSAFE 表示它们不适合承载依赖“这次更新一定会提交”的逻辑，新代码不应使用。

函数组件：Effect 的完整生命周期

useEffect(() => {
  // setup：连接外部系统
  const connection = connect(roomId);

  return () => {
    // cleanup：撤销上一次 setup 做的事情

roomId 改变先执行上一次 cleanup → 再执行新的 setup旧 cleanup 读取旧值；新 setup 读取新值
依赖没有变化setup 和 cleanup 都不执行—

依赖数组控制同步频率：不传依赖数组时，每次提交后执行；传 [] 时只在挂载后 setup、卸载时 cleanup；传 [a,b] 时，任一依赖通过 Object.is 比较发生变化才重新同步。

为什么 Effect 不能只理解为“挂载后执行”

假设聊天室组件当前连接 general 房间，用户随后把 roomId 改成 travel。组件没有卸载，但旧连接必须断开，并建立新连接：

① roomId = generalEffect setup：连接 general。
② 用户选择 travel组件重新渲染，界面显示 travel。
③ 先执行旧 cleanup断开 general，避免同时保留两个连接。
④ 再执行新 setup连接 travel，使外部连接与当前 props 一致。

因此，一个 Effect 更适合看成一对操作：setup 建立某种外部连接，cleanup 撤销这次连接。依赖变化时，这一对操作会重复发生；它不只对应挂载和卸载。

需要 Effect

useEffect(() => {
  const c = connect(roomId);
  return () => c.disconnect();
}, [roomId]);

聊天室连接是 React 之外的系统，需要在 props 改变后重新同步。

不需要 Effect

// 直接由本次 props 计算
const total = price * quantity;

total 只是已有数据的计算结果，不需要先渲染一次，再通过 Effect 写回 state。

StrictMode 下为什么会多执行一次

React 额外执行一次，是为了让两类原本偶发的 bug 在开发时立即出现。正确的组件应该满足：

  • 组件函数执行两次，结果仍相同。组件不能在 Render 中修改 props、全局变量或其他外部数据。
  • Effect 可以安全地 setup → cleanup → setup。cleanup 必须完整撤销上一次 setup。

检查 1：Render 是否纯净

function List({ items }) {
  items.push('新项目'); // ❌ 修改了 props 数组
  return items.map(...);
}

普通执行一次时，错误可能不明显。StrictMode 连续执行两次后，“新项目”会重复出现，直接暴露 Render 修改了外部数据。

检查 2：Effect 是否正确清理

useEffect(() => {
  socket.connect();
  // ❌ 没有 return disconnect
}, []);

StrictMode 执行 setup → cleanup → setup。因为 cleanup 缺失，最终会保留两个连接，问题立刻可见。补上 return () => socket.disconnect() 后，任意时刻只保留一个连接。

为什么模拟“建立 → 清理 → 再建立”？真实应用中，用户可能进入页面、离开页面、再回来；依赖变化时 Effect 也会反复清理和建立。StrictMode 提前做一次压力测试，确认代码能承受这个过程。这项检查仅在开发环境执行，生产环境不会因此多渲染一次页面。

9 · 双缓存 Fiber 树

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

React 同时维护两棵 Fiber 树——current(屏幕上显示的)和 workInProgress(内存里构建的),靠 alternate 互指,像双缓冲。

本章包括：① 两棵树什么关系 ② 怎么交替切换 ③ 为什么需要它 ④「复用」到底复用了什么。

React 同时维护两棵 Fiber 树。它们不是两棵毫无关系的树:两棵树上「同一个位置」的节点,是同一个组件的两个版本(两个 fiber 对象),彼此用 alternate 指针互指——就像双缓冲绘图里的「前台画面」和「后台画布」。

先理解 alternate:current 树的节点 A 和 WIP 树里对应的节点 A',满足 A.alternate === A' 且 A'.alternate === A。它们代表同一个组件,只是一个是「现在屏幕上的样子」、一个是「正在算的新样子」。构建 WIP 时,React 会尽量复用 current 节点的 alternate(已有的 fiber 对象),而不是每次都 new。
current 树(屏幕正显示) workInProgress 树(内存中构建) alternate 互指(同一 fiber 的两个版本)
怎么操作：点击“下一步”逐个构建右侧 WIP 树。重点看黄色虚线如何连接同一 Fiber 的两个版本，以及最后 fiberRoot.current 如何从左树切到右树。
初始:current 树(蓝)在屏幕上,fiberRoot.current 指向它;WIP 尚未构建。点「下一步」。

三个关键点

  • 交替:每次更新,以 current 为蓝本在内存构建 workInProgress。构建完,fiberRoot.current 指针从旧树切到新树 —— 一次指针切换 = 整屏更新。
  • 复用:WIP 节点尽量复用 current 上的 alternate(已有的 fiber 对象),减少创建和 GC。旧树不丢,留作下次画布。
  • 为什么需要它:有两棵树,才能在内存里安全构建 WIP 的同时,current 保持完整可用。被中断时直接丢弃 WIP,current 不受影响,用户永远看到一致画面。这正是可中断渲染的根本。

「复用」到底复用了什么?(createWorkInProgress)

先以一个普通组件 Fiber 为例，把两个配对对象叫作 A 和 B。React 更新时不是一直修改 current，也不是每次创建第三个对象，而是让 A、B 轮流充当 current 和 workInProgress：

时间current（屏幕对应的已提交结果）workInProgress（正在计算）发生什么
首次挂载完成A还没有A.alternate = null
第一次更新AB（首次创建）A、B 通过 alternate 互相指向
第一次提交后BA 留作下次使用root.current 从 A 切到 B
第二次更新BA（直接复用）重置 A 的临时字段，在 A 上计算新结果
因此，复用 Fiber 对象指的是复用另一棵树中的配对对象。第一次更新必须创建 B；从第二次更新开始，current.alternate 已经存在，只需清理上次计算留下的临时字段，然后再次使用。

先看真正决定“创建还是复用”的代码

function createWorkInProgress(current, pendingProps) {
  let wip = current.alternate;

  if (wip === null) {
    // 第一次更新：配对 Fiber 还不存在，只创建这一次
    wip = createFiber(current.tag, pendingProps, current.key);
    wip.stateNode = current.stateNode;
    wip.alternate = current;
    current.alternate = wip;
  } else {
    // 后续更新：直接拿已有的配对 Fiber
    wip.pendingProps = pendingProps;
    wip.flags = NoFlags;
    wip.subtreeFlags = NoFlags;
    wip.deletions = null;
  }

  return wip;
}

上面的 else 就是“复用对象”的真实位置：没有调用 createFiber，而是拿到旧的 A 或 B，清掉上次 Render 留下的 flags，再把它当作新的工作区。

为什么还要从 current 复制字段

复用的是对象外壳，但本轮计算必须以上一次提交结果为起点，所以函数后半段还会把 current 的基线数据放到 wip：

// 本轮计算的起点
wip.child = current.child;
wip.memoizedProps = current.memoizedProps;
wip.memoizedState = current.memoizedState;
wip.updateQueue = current.updateQueue;
wip.lanes = current.lanes;
wip.childLanes = current.childLanes;
字段为什么先带到 wip本轮之后会怎样
stateNode继续指向同一个 DOM 节点或类组件实例属性变化时修改该 DOM，不重新创建
memoizedState以上次提交的 state / Hook 链表为计算起点处理 update 后，wip 保存本轮算出的新状态
memoizedProps用于和新传入的 pendingProps 比较Render 完成后记录本轮已经使用的 props
child先取得旧子树，供 Diff 判断哪些子 Fiber 可以复用Diff 后指向本轮得到的新子 Fiber
把“复用”拆成两层就清楚了：第一层，A/B 两个 Fiber 对象轮流作为工作区，避免反复创建对象；第二层，本轮的 wip Fiber 仍指向原来的 stateNode，并以上次 state 为计算起点，所以 DOM 和组件状态能够延续。只有 type 或 key 不匹配时，旧 Fiber 才会删除，并挂载新的 Fiber、DOM 和 state。
Render 的两棵树：current 保存已提交界面，workInProgress 用于本次计算；Render 被中断时不会影响 current。下一章看 Hooks 如何在执行组件时读取并计算状态。

11 · Diff：协调子节点

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

Diff 就是「拿新描述去更新/复用旧 fiber」;靠三大假设把最优算法的 O(n³) 砍到 O(n)。

本章包括：三大假设 → 单节点 Diff → 多节点两轮遍历 → lastPlacedIndex 移动判断 → index 当 key 为何出 bug。

对比两棵任意树的最优算法是 O(n³)(n 是节点数)——对上千节点的页面完全不可用。React 放弃「找到最优最少操作」,转而基于三条符合前端实际情况的假设(启发式策略),把复杂度砍到 O(n)。理解这三条假设,就理解了 React Diff 的一切行为(包括各种「诡异」的复用/重建)。

三大假设(Diff 的地基,务必吃透)

🧮 算法复杂度优化

React 通过三大启发式假设,把 Diff 从理论上的 O(n³) 降到 O(n)——用线性时间复杂度实现高效的虚拟 DOM 比对。

💡 三大核心假设

1
跨层级节点移动极少发生
只在同一父容器的直接子节点范围内做 Diff(跨层就删了重建)
2
不同 type 的节点生成不同的树
type 不同则跳过子树比对,直接卸载旧的、重建新的
3
同层子节点用 key 标识稳定身份
通过 key 建立「节点 ↔ 数据」的稳定关联,才能移动复用

下面逐条展开(每条都有代码示例 + 实战陷阱):

①只比同层级,不跨层移动 Tree Diff

React 只在同一层级、同一父节点下对比新旧节点,绝不跨层级比较或移动。如果一个节点在更新后「换了父亲/换了层级」,React 不会去「找到它并搬过去」,而是:在旧位置删除它、在新位置重新创建。

// 更新前                    更新后(把 A 从顶层挪进 B 里)
<div><A/><B/></div>           <div><B><A/></B></div>
// React 不会「移动」A,而是:删掉顶层的 A、在 B 里新建一个 A
// → A 的 state 丢失、DOM 重建。所以:别用跨层级移动来做动画/结构变化
为什么敢这么假设?因为 Web UI 里跨层级移动节点极少见。为了这种罕见情况做「全树查找匹配」不划算,直接删了重建,换来 O(n) 的同层遍历。

②type 不同,整棵子树直接换掉 Component Diff

对比同一位置的新旧节点时,只要 type 不一样(div→span、组件 Foo→Bar),React 立即认定整棵子树都变了:卸载旧子树(触发 unmount / componentWillUnmount、清理 effect)、从零构建新子树,不再深入对比里面的子节点。只有 type 相同,才保留该节点、只更新变化的属性,并继续 Diff 它的孩子。

// type 相同 → 复用节点,只 patch 属性,继续 diff 子树
<div className="a">  →  <div className="b">   ✅ 复用 div,只改 class

// type 不同 → 整棵子树销毁重建(哪怕内部长得几乎一样)
<div>…大量子节点…</div>  →  <section>…相同子节点…</section>
// div≠section → 里面的子节点全部卸载重建,state/DOM 全丢
实战陷阱:条件渲染时若外层 type 变了(如 {cond ? <div>… : <section>…}),里面组件会被卸载重挂载、state 丢失。想保留状态,就保持 type 一致(用同一标签 + 条件改属性),或用 key 控制。

③用 key 给同层子节点一个稳定「身份证」 Element Diff

对同层的一组子节点(列表),React 靠 key 判断「新旧两轮里,哪个是同一个节点」。有稳定 key,即使节点在列表里换了位置,React 也能认出它、移动并复用(保留 state 和 DOM),而不是重建。没有 key(或用会变的 index),React 只能按位置硬匹配,顺序一变就全判错。

// 有稳定 key:React 认得出 b 只是移到了前面 → 移动复用
[a,b,c]  →  [b,a,c]     key=id:只移动,不重建 ✅

// 无 key / index 当 key:按位置比 → 以为 位置0 从 a 变成了 b …全部误判
三项前提：① 只看同层(跨层就删了重建)→ ② type 变就整棵换(type 同才复用)→ ③ 同层列表靠 key 认身份(有稳定 key 才能移动复用)。下面「单节点 / 多节点 Diff」就是这三条假设的具体落地。

单节点 Diff(reconcileSingleElement)

更新后只有一个子节点(<div>{x}</div>)时,React 拿这个新元素去和旧的一串子 fiber逐个比:

function reconcileSingleElement(returnFiber, currentFirstChild, element){
  let child = currentFirstChild;
  while(child !== null){
    if(child.key === element.key){          // ① 先比 key
      if(child.type === element.type){        // ② key 同,再比 type
        deleteRemainingChildren(returnFiber, child.sibling); // 其余兄弟全删
        const existing = useFiber(child, element.props); // ★复用!
        return existing;
      }
      deleteRemainingChildren(returnFiber, child); // key 同 type 不同 → 从它起全删
      break;
    } else {
      deleteChild(returnFiber, child);        // key 不同 → 删这一个,继续看兄弟
    }
    child = child.sibling;
  }
  return createFiberFromElement(element);    // 没匹配到 → 新建
}
注意顺序:key 相同但 type 不同时,不仅删自己,还 deleteRemainingChildren 把后面兄弟一起删(因为 key 唯一,后面不可能再有同 key 的);key 不同只删当前继续找。这正是假设 ②③ 的落地。

多节点 Diff:两轮遍历(reconcileChildrenArray)

先不要看源码，固定一个例子：旧列表是 [A, B, C, D]，新列表是 [A, C, E, B]。字母就是稳定的 key，元素 type 都相同。

第一轮：只比较同一个下标，直到第一次对不上

新下标旧节点新节点结果
0AAkey/type 相同，复用 A；继续
1BCkey 不同，第一轮立即停止

停止时的现场：A 已处理；旧节点还剩 [B,C,D]；新节点还剩 [C,E,B]。React 不会继续拿旧 C 和新 E 做同位比较，而是转入第二轮。

第二轮：旧节点建 Map，新节点按 key 查找

先把剩余旧节点建成：{ B → oldB, C → oldC, D → oldD }。

依次处理的新节点去 Map 查找React 做什么Map 剩余
C找到 oldC复用 C；从 Map 删除 CB、D
E找不到创建新 Fiber E，并标记 PlacementB、D
B找到 oldB复用 B；根据旧位置判断是否移动D
新数组结束Map 仍有 D旧 D 没被使用，标记删除空
两轮不是把整棵列表重复比较两遍。第一轮只快速吃掉开头连续对得上的部分；一旦错位，第二轮只处理“双方剩下的部分”，用 Map 避免反复扫描旧列表。

更新后是一个数组子节点(列表)时最复杂。React 用两轮遍历(不是 Vue 的严格双端 Diff,但吸收了同类思想):

  • 第一轮:新旧数组从左到右同位置比较,key+type 相同就复用,一旦不同立即跳出。这一轮吃掉最常见的「尾部追加 / 只改内容不换位」。
  • 第二轮:把剩余旧节点存进 Map(key → fiber),遍历剩余新节点用 key 去 Map 里捞。捞到 = 复用(可能要移动);捞不到 = 新建;最后 Map 里没被捞走的 = 删除。

📊 两轮遍历流程总览

lastPlacedIndex:决定「移动还是不动」的灵魂

lastPlacedIndex 记录“目前已处理的可复用节点中，旧位置最靠右的下标”。每处理一个新节点，placeChild 用它判断该节点是否需要移动：

function placeChild(newFiber, lastPlacedIndex, newIndex){
  newFiber.index = newIndex;
  const current = newFiber.alternate;      // 复用的旧 fiber(新建则为 null)
  if(current !== null){
    const oldIndex = current.index;        // 它在旧列表里的原位置
    if(oldIndex < lastPlacedIndex){         // ★原来在左边,现在要在右边 → 得移动
      newFiber.flags |= Placement;         // 标记「移动」
      return lastPlacedIndex;
    } else {
      return oldIndex;                     // 原来就靠右 → 不用动,抬高基准线
    }
  } else {
    newFiber.flags |= Placement;           // 新建也是 Placement(插入)
    return lastPlacedIndex;
  }
}

要点：只有当一个可复用节点「原来的位置比基准线还靠左」时,才判定它要移动;否则不动、并把基准线抬到它的位置。这是一种单向(只往右)的贪心,不做 Vue 那种双端对比。

互动:lastPlacedIndex 追踪器(逐步看谁要移动)

怎么操作：先选择一种新顺序，再点“下一步”。每一步只看两个数字：当前节点的 oldIndex 与 lastPlacedIndex；前者更小时，该节点会标记为移动。
选一个变换,然后单步执行。
惊人但真实的结论:把最后一个元素移到最前(D,A,B,C)会让 A、B、C 三个都被标记移动(3 次 DOM 移动);而把第一个移到最后(B,C,D,A)只移动 A 一个。 因为 lastPlacedIndex 是「只往右」的贪心——它对「向后移动」友好、对「向前移动」很贵。所以做列表头部插入 / 逆序这类操作时,即使 key 正确,也可能产生大量 DOM 移动。这是 React Diff 一个真实的性能特性。
React 18 优化点:Diff 算法本身自 16 起就稳定(仍是上面这套两轮 + lastPlacedIndex)。18 的性能收益来自 Concurrent(把大 Diff 切片摊到多帧、可中断)+ 自动批处理(减少 Diff 次数),不是改了 Diff 算法。

10 · Hooks：读取与计算状态

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM

先想清楚 Hooks 到底解决什么问题,再谈底层。你已经知道(第一层·组件渲染):函数组件每次渲染都从头到尾重新执行一遍。那么——

灵魂拷问:函数一跑完,里面的局部变量就都销毁了,可为什么下次渲染时 useState 的 count 还在、还记得上次的值?

答案:React 把状态存在函数外面——存在这个组件对应的 Fiber 节点上。所谓 Hook,就是「在 Fiber 上开一块记忆、并把它和你的每次调用对应起来」的机制。整章就回答三个问题:

  1. 存在哪? 状态放在 Fiber 的哪个位置(下面 ①)
  2. 怎么对应上? 重新渲染时,React 怎么知道哪个值属于哪个 Hook(② ③)
  3. 每种 Hook 具体存什么? useState / useEffect / useMemo 各放什么(④)

① 存在哪:Fiber 上的一排「储物柜」

把 Fiber 想成一排编号的储物柜:你在组件里调用的第 1、2、3 个 Hook,就对应 1 号、2 号、3 号柜子。技术上,这排柜子是挂在 Fiber 的 memoizedState 字段上的一条单链表,每个 Hook 是一个节点,靠 next 串起来:

// 每个 hook 节点
{
  memoizedState,  // 该 hook 存的值(state / effect / memo 值…)
  baseState,      // 并发中断时的「存档起点」
  baseQueue,      // 被跳过、留待下次的更新
  queue,          // 更新队列(useState / useReducer)
  next,           // → 下一个 hook
}
fiber.memoizedState → hook1 → hook2 → hook3 → null

② 怎么对应上:只认「第几个」,所以不能条件调用

存进链表了,那 React 重新渲染时怎么把值对回每个 Hook?关键点:它不看 Hook 叫什么名字,只数「你调用的第几个」——第 1 个调用拿 1 号柜、第 2 个拿 2 号柜……首次渲染按顺序建柜子,以后每次渲染按同样的顺序挨个取。

所以一旦你 if(x) useState() 少调了一个,后面所有 Hook 的「第几个」就全部错位一格,值张冠李戴。铁律:每次渲染,Hook 的数量和顺序必须完全一致(不能放进 if / for / return 之后)。下面这个演示把「按位置对应」画出来了:

③ 同一个 useState，为什么首次和更新走不同代码

组件代码里始终写的是 useState(initialState)，但 React 内部需要处理两种不同情况：

首次 Render：还没有 Hook 节点

调用 mountState，创建 Hook 节点，保存初始 state 和更新队列，并把节点接到当前 Fiber 的 Hook 链表。

更新 Render：Hook 节点已经存在

调用 updateState，按调用顺序取得对应 Hook，处理它队列中的 update，计算本轮 state。

公开的 useState 函数并不知道当前是首次还是更新。函数组件开始 Render 时，renderWithHooks 会先把当前实现表设置为 mount 版或 update 版；随后 useState 从这张表中调用对应函数。这张“当前实现表”就是 dispatcher。

组件执行到 useState()
↓ 读取当前 dispatcher 中的 useState 实现
首次 Render → mountState
创建 Hook，保存初始值
更新 Render → updateState
取得旧 Hook，处理更新队列

对应到源码，就是在执行函数组件之前选择一套实现，然后让公开 API 转发过去：

// renderWithHooks:每次渲染函数组件前,按 mount/update 换 dispatcher
ReactCurrentDispatcher.current =
  current === null || current.memoizedState === null
    ? HooksDispatcherOnMount     // 首次:mountState / mountEffect …(创建 hook)
    : HooksDispatcherOnUpdate;   // 更新:updateState / updateEffect …(复用 hook)

// useState 转发到当前 dispatcher
function useState(initial){
  return resolveDispatcher().useState(initial);
}
为什么组件外调用 Hook 会报错？因为组件外没有正在执行的 renderWithHooks，也就没有可供本次 Render 使用的 Hook dispatcher。React 无法知道应该把 Hook 节点挂到哪个 Fiber。

④ 各 Hook 具体存什么 · useState / useReducer

前面讲的是通用机制(链表、按序对应)。现在看每种 Hook 的「柜子」里到底放了什么。

useState 复用 useReducer 的更新流程，并使用内置的 state reducer（(s,a)=>typeof a==='function'?a(s):a）。dispatch 时，update 进入该 Hook 的 queue.pending 环状链表（详见Trigger·UpdateQueue）；渲染时从 baseState 开始依次处理 action，得到新的 memoizedState。

// dispatchSetState 里的「提前计算」优化(eager state bailout)
if(fiber.lanes === NoLanes){          // 队列里没有别的待处理更新时
  const eager = reducer(currentState, action); // 先算出新 state
  update.hasEagerState = true; update.eagerState = eager;
  if(is(eager, currentState)) return;  // 新旧相等 → 直接不调度渲染!
}
scheduleUpdateOnFiber(fiber, lane);   // 不相等才真正安排渲染
这就是「setCount(1) 时 count 已经是 1 就不重渲染」的底层原因:dispatch 时先用 reducer 试算,若结果和当前 state Object.is 相等,连渲染都不安排(注意:若是父级传下的其它更新导致重渲染,该组件仍可能渲染)。

④ 各 Hook 具体存什么 · useEffect(副作用)

useEffect 的 hook 节点里存一个 effect 对象 {tag, create, destroy, deps, next},所有 effect 再单独串成一条环状 effect 链表挂在 fiber 上。Render 阶段对比 deps 决定是否给 fiber 打 Passive flag;Commit 后异步遍历这条链表执行(详见Render·变更标记)。

④ 各 Hook 具体存什么 · useMemo / useCallback(缓存)

// useMemo 简化:hook.memoizedState = [value, deps]
function useMemo(create, deps){
  const hook = updateWorkInProgressHook();
  const prev = hook.memoizedState;
  if(prev !== null && deps !== null && areHookInputsEqual(deps, prev[1]))
    return prev[0];              // deps 没变 → 直接返回缓存值
  const next = create();          // 变了 → 重算
  hook.memoizedState = [next, deps];
  return next;
}
// useCallback 只是把函数本身当值缓存:hook.memoizedState=[fn,deps]

Hook它的 memoizedState(柜子)里存的东西
useState / useReducer当前 state 值 + 一个更新队列 queue
useEffect / useLayoutEffect一个 effect 对象 {create, destroy, deps, tag}
useMemo[缓存的计算结果, deps]
useCallback[缓存的函数, deps]
useRef一个 { current } 对象(见 补充·useRef)
useContext不占柜子(readContext 不走 hook 链表,只登记依赖)

🗺 本章路线图

Fiber 既是一种数据结构(带指针的对象),也是一个可拆分、可中断的工作单元。

本章包括：① Fiber 字段含义 ② tag(节点类型)③ beginWork/completeWork 各干什么(深度优先遍历)。

Fiber 节点字段含义

const fiber = {
  // —— 身份:我是谁 ——
  tag,            // 类型标记(FunctionComponent/HostComponent/HostRoot...)
  key,            // 元素的 key
  type,           // 元素类型(组件函数 / 'div')
  stateNode,      // 关联实例:DOM 节点 / class 实例 / fiberRoot

  child,          // 第一个子 fiber
  sibling,        // 下一个兄弟 fiber
  index,          // 在兄弟中的位置(多节点 Diff 用)
  // —— 工作数据 ——
  pendingProps,   // 新 props(本次待处理)
  memoizedProps,  // 上次渲染用的 props
  memoizedState,  // class:state / 函数:hooks 链表头

  // —— 调度 & 副作用 ——
  lanes,          // 本 fiber 的优先级车道

  alternate,      // 指向另一棵树里的自己
};

tag:Fiber 到底是「哪种」节点

tag 表示 Fiber 的节点类型，beginWork 和 completeWork 会据此选择处理逻辑。常见取值如下：

tag对应说明
HostRoot(3)根节点整棵树的顶,stateNode 指向 fiberRoot
FunctionComponent(0)函数组件渲染 = 调用函数;hooks 挂 memoizedState
ClassComponent(1)类组件stateNode 是实例;state 挂 updateQueue
HostComponent(5)原生标签(div/span)stateNode 指向真实 DOM
HostText(6)文本节点纯文本
Fragment / ContextProvider / SuspenseComponent …特殊节点各有专门的 begin/complete 处理

括号里的数字（如 HostRoot(3)）是 tag 的实际值。FunctionComponent、HostRoot 和 HostComponent 是 ReactWorkTags.js 中定义的数字常量，因此源码可以通过 switch(fiber.tag) 区分节点类型。

工作单元:可拆分、可中断

每个 Fiber 就是一个"工作单元"。整棵树的协调 = 循环处理一个个工作单元,循环在单元之间可被 shouldYield() 打断(详见第四层·Concurrent)。

beginWork 与 completeWork:各干一件事

Render 阶段主要通过下面两个函数遍历 Fiber 树：

beginWork(往下)

输入一个 fiber → 算出它的「子节点」。

怎么算:如果是函数组件,就调用你写的函数拿到返回的 JSX;然后把新旧子节点做 Diff,生成子 fiber。

产物 = 子 fiber(带着往下走)。它不碰真实 DOM。

completeWork(往上)

输入一个 fiber → 为它造好「真实 DOM」。

怎么造:document.createElement 建节点,把已经造好的子 DOM 塞进去,设置属性。

产物 = 一个离屏 DOM 节点(带着往上走)。

「把新旧子节点做 Diff,生成子 fiber」到底啥意思? 先分清两个东西:你 return 的 JSX(如 <h1>hi</h1>)只是这次想要的「描述」,还不是 fiber;而上次渲染已经为它生成过对应的旧子 fiber(存在 current 树上,带着真实 DOM 和 state)。beginWork 拿新描述和旧 fiber 逐个对比(这就是 Diff):
· 新旧是同一个(key+type 相同)→ 复用旧 fiber,只改属性。例:上次 <h1>hi</h1>、这次 <h1>bye</h1> → 还是 h1,复用那个 h1 fiber(连同它的 DOM),只把 hi 改成 bye,打个 Update;
· 变了(如 <h1> 变 <p>)→ 删掉旧的、新建一个;新增/减少同理。
目的:能复用就复用,别每次把 fiber 和它的 DOM/state 全部重造。产物就是这次的「子 fiber」(有的复用、有的新建)。「复用」具体复用了什么(stateNode/memoizedState、createWorkInProgress)→ 见 Render·双缓存。

光看定义还是抽象,连起来走一个最小例子最清楚。渲染这个组件:

① beginWork(App)      // 调用 App() → 得到 <h1>hi</h1>,生成 h1 的 fiber   往下 ↓
② beginWork(h1)       // h1 的子是文本 "hi",生成文本 fiber              往下 ↓

点开：beginWork / completeWork 的源码骨架（读源码时看）

1) 谁在驱动遍历——performUnitOfWork 调 beginWork,拿到子节点就往下;没子节点就交给 completeUnitOfWork 往上走(靠 sibling / return 指针):

function performUnitOfWork(wip){
  const next = beginWork(wip.alternate, wip, renderLanes);
  if(next === null) completeUnitOfWork(wip); // 没子节点 → 往上归
  else        workInProgress = next;              // 有子节点 → 往下递
}
function completeUnitOfWork(wip){
  let node = wip;
  do {
    completeWork(node.alternate, node, renderLanes);
    if(node.sibling){ workInProgress = node.sibling; return; } // 有兄弟 → 去兄弟
    node = node.return; workInProgress = node;                    // 没兄弟 → 回父节点
  } while(node);
}

2) beginWork 先判断这个 Fiber 是否真的需要重新计算。如果它的 props、state 和 context 都没有需要处理的变化，React 会直接复用上一次的 Fiber 子树，不再执行组件和 Diff。源码把这种“确认没变化后提前退出”的优化叫作 bailout。

可以先把 bailout 理解成“这条分支没变，跳过”。如果不能跳过，beginWork 才会根据 tag 判断 Fiber 类型：函数组件就执行函数和 Hooks，类组件就调用 render，原生标签就读取 props.children；随后通过 reconcileChildren 比较子节点。
function beginWork(current, wip, renderLanes){
  switch(wip.tag){
    case FunctionComponent: return updateFunctionComponent(…); // renderWithHooks:调你的函数、跑 Hooks
    case ClassComponent:    return updateClassComponent(…);    // 调 instance.render()
    case HostRoot:          return updateHostRoot(…);          // processUpdateQueue 取 <App/>
    case HostComponent:     return updateHostComponent(…);     // 取 props.children
  } // 每个分支最后都 reconcileChildren(做 Diff)并 return wip.child
}

3) completeWork 内部:首次挂载建 DOM 并把子 DOM 拼进去;更新则算属性差异打 Update;最后 bubbleProperties 把子树副作用冒泡进 subtreeFlags(即Render·变更标记的收集):

function completeWork(current, wip){
  switch(wip.tag){
    case HostComponent:
      if(current && wip.stateNode){
        updateHostComponent(current, wip, …);   // 更新:diffProperties 算差异,打 Update
      } else {
        const dom = createInstance(type, props); // 首次:建真实 DOM
        appendAllChildren(dom, wip);             // 把已造好的子 DOM 塞进去
        finalizeInitialChildren(dom, type, props); // 设初始属性
        wip.stateNode = dom;
      }
      break;
  }
  bubbleProperties(wip); // 汇总子树 flags → subtreeFlags
}

5 · UpdateQueue：更新如何入队

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

一个组件 = 一个 Fiber = 一个 UpdateQueue;setState 把更新排进队列,渲染时一起 reduce 出最终 state。

本章包括：① update 长啥样 ② 怎么入队(环状链表)③ 怎么算(processUpdateQueue)④ baseState/baseQueue(并发存档)。

每次状态更新都会进入对应的更新队列。不过，类组件/HostRoot 与 useState Hook 使用的队列字段并不相同。它们的共同目标都是：保存新更新、按插入顺序计算，并在跳过低优先级更新后保留重算所需的信息。

① 先区分类组件队列与 Hook 队列

// 一个 Update 对象(class 组件的形态;hook 版字段略有差异)
const update = {
  lane,              // 本次更新的优先级车道
  payload,           // setState 传的对象,或 (state)=>newState
  next,              // → 下一个 update(环状链表)
};

类组件与 HostRoot

fiber.updateQueue = {
  baseState,                  // 计算的「起点」state
  firstBaseUpdate,            // 基础队列的头
  lastBaseUpdate,             // 同一队列的尾
  shared: { pending: null },  // 新进入的环状队列
}

firstBaseUpdate 与 lastBaseUpdate 不是两个队列，而是同一条单向链表的头、尾两个指针。

useState / useReducer Hook

hook = {
  baseState,
  baseQueue,                  // 基础环状队列的尾指针
  queue: {
    pending: null             // 新进入的环状队列
  }
}

Hook 没有 firstBaseUpdate/lastBaseUpdate；它用一个 baseQueue 尾指针表示基础环状队列，队头同样是 baseQueue.next。

为什么类组件要同时保存 first 和 last？处理队列时要从 first 开始向后遍历；合并新 update 时又要快速找到 last 追加。只有 first，追加时需要遍历整条链表；同时保存 last，追加就是 O(1)。

② 新 update 先进入 pending 环状链表

下面先看类组件的 shared.pending；Hook 的 queue.pending 插入方式相同。多次更新形成环状链表，pending 指向最后一个节点，pending.next 指向第一个节点。

先把范围说准确：“pending 环转成普通链表”特指类组件与 HostRoot。Hook 的 queue.pending 也是环，但它会与同为环状的 baseQueue 合并，处理时仍可用“记住队头，绕一圈回到队头即结束”的方式遍历。

为什么同一批 update 要切换两种形态

因为这批 update 在进入 Render 前后承担的是两种完全不同的任务。可以把 pending 看成收件箱，把基础队列看成 Render 的工作清单：

所处阶段主要操作适合的形态为什么
事件发生时：收集不断把新 update 追加到末尾尾指针表示的环只存 pending 一个指针，就能用 pending 找队尾、用 pending.next 找队头；每次追加都是 O(1)。此时不需要遍历或计算 state。
Render 开始后：处理从最早的 update 向后计算，并把跳过的 update 保存成新的基础队列类组件使用普通单链表从 firstBaseUpdate 一路走到 null 即结束；同时用 lastBaseUpdate 快速追加，适合顺序消费和重建基础队列。

环转普通链表时，具体只发生四件事

  1. 先保存队尾：last = shared.pending，例如得到 u3。
  2. 在断环前保存队头：first = last.next，因此仍能找到 u1。
  3. 断开队尾到队头的回边：last.next = null，于是 u1 → u2 → u3 → u1 变为 u1 → u2 → u3 → null。
  4. 接入工作清单：把 first/last 追加到类组件基础队列的 firstBaseUpdate/lastBaseUpdate 后面。
为什么一定要先读 last.next 再断环？因为外部只保存了队尾 last，队头藏在 last.next 中。如果先执行 last.next = null，就会丢失队头入口，无法找到 u1。正确顺序是“先取队头，再断开回边”。

所以，环并不是“计算 update 更方便”，普通链表也不是“入队更快”。React 是在每个阶段选择更适合当下操作的表示：环负责高频收集和整批移交，普通链表负责有明确终点的顺序计算。

为什么收集阶段选择尾指针环

pending 可能连续插入很多次，但一轮 Render 通常只需接管这个入口。环状结构让收集阶段只保存一个 pending 指针，却能同时 O(1) 找到队尾和队头：

环的收益发生在入队阶段：假设 Render 前来了 100 个 update，React 会做 100 次 O(1) 入队；Render 接管时只做一次 O(1) 的断环。总成本仍是 O(n)，但不会出现“第 1 次找 0 个节点、第 2 次找 1 个节点……第 100 次找 99 个节点”的反复尾部遍历。
null 终点为什么适合类组件的处理过程？processUpdateQueue 不只是读取旧队列，还会在遇到低优先级 update 后构造一条新的基础队列。普通链表可以用 update !== null 控制遍历，并用新的 first/last 指针逐个追加需要保留的节点；无需额外保存“绕一圈后的停止哨兵”。

普通的 head + tail 双指针链表也能实现 O(1) 入队；环状链表不是唯一方案。React 的选择是用一个尾指针同时编码头和尾，并让 Hook 的 pending 环与 baseQueue 环可以通过少量指针修改完成合并。

连续插入 u1、u2、u3 后，指针怎样变化

最终只要记住两个入口：pending 是队尾 u3；pending.next 是队头 u1。遍历从 u1 开始，依次经过 u2、u3，再回到 u1 时结束。
function enqueueUpdate(fiber, update){
  const queue = fiber.updateQueue.shared;
  const pending = queue.pending;
  if(pending === null) update.next = update;    // 第一个:自己成环
  else { update.next = pending.next;             // 新节点接到头前
         pending.next = update; }
  queue.pending = update;   // pending 永远指向「最后进来的」
}
把源码代入“插入 u3”：
进入函数前：pending = u2，pending.next = u1。
① u3.next = pending.next，所以 u3.next = u1；
② pending.next = u3，所以 u2.next = u3；
③ queue.pending = u3，队尾指针从 u2 移到 u3。
旧的 u1.next = u2 不需要修改，最终得到 u1 → u2 → u3 → u1。

Hook 为什么也保留环：两个队列可以 O(1) 拼接

Hook 的 baseQueue 和 queue.pending 都是“尾指针 + 环”。假设旧基础队列是 b1 → b2 → b1，新 pending 是 p1 → p2 → p1。React 已经同时拿到了两个队尾，因此只要交换两条连接：

const baseFirst = baseLast.next;        // b1
const pendingFirst = pendingLast.next;  // p1

baseLast.next = pendingFirst;               // b2 → p1
pendingLast.next = baseFirst;               // p2 → b1，重新闭环
baseQueue = pendingLast;                     // 新队尾是 p2
合并后：b1 → b2 → p1 → p2 → b1。旧更新仍在前，新更新仍在后；无论两个环里各有多少节点，合并都不需要遍历，所以仍是 O(1)。这比“环能少存一个指针”更重要：它让 React 能快速接管、拼接和保留整批更新。

④ 渲染时怎么算:从 baseState 逐个 reduce(processUpdateQueue)

渲染时 React 调用一个叫 processUpdateQueue 的函数来算最终 state:从 baseState(起点值)出发,按顺序把队列里每个 update 应用一遍(像滚雪球),得到本次的 memoizedState。这就是「多次 setState 被合并成一次结果」:

// processUpdateQueue 概念:从 baseState 开始,依次 apply 每个 update
let newState = queue.baseState;
for(const u of 队列里的 update){
  newState = reduce(newState, u);   // u 是对象则合并;是函数则 u(newState)
}
fiber.memoizedState = newState;      // 得到本次最终 state

所以 setCount(c=>c+1) 连写三次会累加(每个 update 拿到前一个的结果),而 setCount(count+1) 连写三次只 +1(闭包里 count 是同一个旧值)。

⑤ 为什么需要 baseState 和基础队列

先不要记字段名。只看同一个 count 按顺序收到的三次更新：

u1: c => c + 1（高优） → u2: c => c + 10（低优） → u3: c => c * 2（高优）

初始值是 0。React 先处理高优更新时，会按原顺序走一遍：

  1. 执行 u1：0 + 1 = 1。
  2. 暂时跳过 u2：它是低优更新。这一刻的值 1 被记为 baseState，意思是“下次补算要从这里开始”。
  3. 执行 u3：1 × 2 = 2，所以本次可以先显示 2。

问题是：以后补算 u2 时，不能只算 1 + 10 = 11。原始顺序是 u2 → u3，所以还要再算一次 u3，最终才是 (1 + 10) × 2 = 22。

基础队列就是“下次必须重放的那一段顺序”。从第一个被跳过的 u2 开始，React 保存 [u2, u3]。u3 本次虽然已经执行，仍要留一份副本，保证下次能恢复原来的计算顺序。

为什么 u3 的副本使用 NoLane？因为 u3 已经通过了本次的优先级检查。下次重放时，它必须跟在 u2 后面执行，不能再因 Lane 不匹配而被跳过；NoLane 在这里表示“这个副本重放时直接执行”。

“无关的更新”要分情况：另一个 useState 有自己的队列，不会混进来；但同一个 state 队列里的后续 update 都要保留顺序。React 不会猜测两个更新函数在业务语义上是否互不影响。

源码中，isSubsetOfLanes(renderLanes, update.lane) 决定本次是否跳过 update。类组件用 firstBaseUpdate/lastBaseUpdate 保存这段基础队列；State Hook 用 hook.baseQueue 保存。

「副作用(effect)」= 组件在纯计算之外要对外界做的事:改真实 DOM、跑 useEffect、更新 ref。

核心结论：Render 阶段在 Fiber 上设置 flags，记录之后需要执行的操作；Commit 阶段读取这些 flags，并修改 DOM 或执行相关副作用。

为什么分两步:Render 可中断、可丢弃重来,不能真去动 DOM;所以先把「要做的事」攒成标记,等 Commit 一次性做完。

本章回答三个问题:① 有哪些标记? ② 标记记在哪、Commit 怎么高效找到它们? ③ 什么时候执行?(第 ③ 步的执行细节在下一章 Commit·提交阶段)

先回答：一个“有变化的 Fiber”怎么产生

不是拿 baseState 直接做 Diff。一次组件更新在 Render 阶段内部按下面顺序发生：

  1. 先算 state：processUpdateQueue 从旧 state 出发处理本轮 update，得到新的 memoizedState。baseState 只在有 update 被跳过时作为未来重算起点。
  2. 再执行组件：组件使用新 state/props 返回新的 React Element，例如从 <h1>0</h1> 变成 <h1>1</h1>。
  3. 再做子节点 Diff：reconcileChildren 比较旧子 Fiber 与新 React Element，决定复用、新建、移动或删除哪些 WIP Fiber。
  4. 再比较 DOM 属性：completeWork 比较宿主 Fiber 的旧 props 与新 props，发现文本、className、style 等差异。
  5. 最后留下 flags：新增/移动产生 Placement，删除产生 Deletion，属性/文本变化产生 Update；effect deps 变化产生 Passive 或 Layout 相关标记。
因此答案是：state 由 UpdateQueue 计算；新 JSX 由组件执行产生；Fiber 结构变化由 Diff 发现；DOM 属性变化由 completeWork 发现。它们不是同一步，但都发生在 Render 阶段，并最终把操作记录到 WIP Fiber。

    setText(e.target.value);              // 紧急：输入框立刻显示
    startTransition(() => {
      setQuery(e.target.value);           // 可延后：重新计算大列表
    });
  }
}

假设列表很大，Transition Render 需要多个时间片。下面所有概念都围绕这件事讲：

① Transition update 入队
setQuery 的 update 带 Transition Lane；root.pendingLanes 记录它。
② React 为 root 安排一个低优任务
ensureRootIsScheduled 把“处理这个 root”的回调交给 Scheduler。
③ Scheduler 执行回调，Fiber Render 开始
React 在 workInProgress 树上逐个计算列表 Fiber，真实 DOM 尚未改变。
④ 时间片用完，React 暂停
workInProgress 指向“下一个还没处理的 Fiber”，Scheduler 安排后续回调。
⑤ 浏览器处理下一次输入
setText 产生 Sync Lane。React 发现它比正在处理的 Transition 更急。
⑥ 先完成 Sync，再重新处理 Transition
低优回调被替换；高优更新先 Render 和 Commit。Transition update 没有消失，随后基于最新 current 重新计算列表。
先记住结果：输入框先更新，慢列表稍后更新。Scheduler 的全部工作，就是让第 ⑤ 步产生的紧急更新不必排在漫长的列表 Render 后面。

Scheduler 实际收到的是什么

Scheduler 只接收“回调函数 + Scheduler 优先级”。它不认识 Lane、root 或 Fiber。React 把 root 放进回调的闭包中，再把这个回调交给 Scheduler：

const callback = performConcurrentWorkOnRoot.bind(null, root);
const task = scheduleCallback(NormalPriority, callback);

// Scheduler 内部看到的近似任务
task = {
  callback,          // 执行它，才会回到 React 并处理这个 root
  priorityLevel,
  startTime,
  expirationTime
};

假设 App 中有一万个 Fiber，React 也不会提交一万个 Scheduler 任务。从 React 的角度看，通常是为一个 root 保留一个并发回调；回调真正执行后，Reconciler 才在内部逐个处理 Fiber。

root.callbackNode      // 当前并发 Scheduler 任务；同步路径可能为 null
root.callbackPriority  // root 当前已安排工作的 Lane 优先级

新的 update 到来后，ensureRootIsScheduled 会再次检查 root：

  • 还没有工作入口：根据 Lane 安排同步入口或并发 Scheduler 任务。
  • 已有同等优先级任务：继续复用，不重复安排。
  • 来了更高优更新：取消旧的并发回调，按更高优先级重新安排。
  • 已经没有待处理 Lane：取消回调并清空 callbackNode。
// ensureRootIsScheduled 的核心判断（简化）
nextLanes = getNextLanes(root.pendingLanes);
newPriority = getHighestPriorityLane(nextLanes);

if (newPriority === root.callbackPriority) return; // 已安排同等工作
if (root.callbackNode !== null) cancelCallback(root.callbackNode);

if (newPriority === SyncLane) {
  scheduleSyncCallback(performSyncWorkOnRoot);    // 同步队列由微任务尽快刷新
  root.callbackNode = null;
} else {
  root.callbackNode = scheduleCallback(
    lanesToSchedulerPriority(nextLanes),
    performConcurrentWorkOnRoot.bind(null, root)
  );
}
  root.callbackPriority = newPriority;
为什么 Sync 能明显插队？Transition 通常等待 Scheduler 的并发回调，并且工作循环会让位；Sync 更新改走同步刷新入口。当前 JavaScript 任务结束进入微任务检查后，React 会尽快完成 Sync Render/Commit。
cancelCallback 取消的只是 Scheduler 任务。它不会删除 update，也不会回滚 DOM，因为低优 Render 根本还没有进入 Commit。update 仍在 Hook/Class 的更新队列中，对应 Lane 仍留在 root.pendingLanes。

情况 A：只是时间片结束，没有更高优更新

workInProgressRootRenderLanes 仍是 Transition，workInProgress 仍指向列表中尚未处理的 Fiber。下一次 Scheduler 回调到来后，React 继续使用同一棵 WIP，从该 Fiber 接着算。这叫暂停和续算，不是抢占。

情况 B：暂停期间来了 Sync 更新

  1. 浏览器处理输入事件，setText 创建 Sync update。
  2. Sync Lane 加入 root.pendingLanes，此时 root 同时有 Sync | Transition。
  3. ensureRootIsScheduled 发现现有低优回调不够急，调用 cancelCallback 取消它；Sync 更新随后进入同步刷新路径。
  4. 高优回调运行时，getNextLanes 选出 Sync。它与当前正在构建的 Transition lanes 不同。
  5. prepareFreshStack(root, SyncLane) 从仍然完好的 root.current 创建新的根 WIP，并重置 workInProgress。先前那棵低优 WIP 不会提交。
  6. Sync Render 遍历更新队列：执行 Sync update，暂时跳过 Transition update，并把后者保留到基础队列。
  7. Sync Commit 后，输入框更新。Transition Lane/update 仍然 pending，于是 React 再安排低优任务，基于最新 current 重算列表。
“丢弃低优 WIP”丢的是计算结果，不是更新请求。半成品 Fiber 上算出的 props/state/flags 没有提交，可以覆盖；原始 Transition update 仍保存在队列中，所以稍后能够重新算出正确结果。

Scheduler 回调怎样驱动 Fiber 工作循环

Scheduler 调用 root 回调后，控制权回到 React Reconciler。Reconciler 准备本次 renderLanes，然后重复处理 Fiber：

function workLoopConcurrent() {
  while (workInProgress !== null && !shouldYield()) {
    performUnitOfWork(workInProgress); // 处理一个 Fiber
  }
}

每个 Fiber 是一个可暂停边界。处理完一个 Fiber 后，React 检查 shouldYield()：

  • 还有时间：继续处理下一个 Fiber。
  • 时间片用完：退出工作循环，保留 workInProgress 指针。
  • Scheduler 再次给 root 执行机会：从这个指针继续。
Fiber 与时间切片的真正关系：Scheduler 只说“现在该暂停了”；Fiber 链表结构让 React 知道暂停在哪里、下次从哪里继续。没有 Fiber 这种工作单元，Scheduler 即使要求让位，React 也很难保存一半的树计算进度。

到底是谁可打断：答案是并发 Render 工作循环

阶段能否时间切片暂停原因
Scheduler 排队/选择任务不叫“React Render 被打断”它只决定何时调用回调，并提供 shouldYield
Concurrent Render可以只修改内存中的 WIP Fiber；每完成一个 Fiber 后可退出循环
Sync Render不会按时间片让位同步工作循环会一直算到完成
Commit不可以正在修改 DOM、ref 和执行 Layout effect，中途停会暴露不一致界面

发生暂停的直接条件：当前是并发 Render，workInProgress !== null，并且处理完一个 Fiber 后 shouldYield() 返回 true。React 退出 workLoopConcurrent，把控制权交回浏览器。

高优更新何时能进来：JavaScript 正在运行时，点击事件不能同时插入执行。必须先由低优 Render 在时间片边界让位，浏览器才有机会派发点击；点击产生高优 update 后，React 下一次选择 lanes 时才发生抢占。

恢复：Lane 没变

保留 workInProgress 指针和部分 WIP 树；下一次回调从原位置继续。

重算：来了更高优 Lane

prepareFreshStack 从 current 建立新 WIP；旧半成品不提交。原 update 仍在队列中，之后重新处理。

Lane 怎样转换成 Scheduler 优先级

Lane 的分类比 Scheduler 更细。ensureRootIsScheduled 先选出 root 下一批要处理的 Lane，再把它映射为 Scheduler 能理解的任务优先级：

React 中的更新Lane 负责Scheduler 负责
点击、键盘等紧急更新选出高优 Lane尽快运行对应 root 回调
普通异步更新选出 Default Lane按普通优先级排入 taskQueue
startTransition 更新标为可延后的 Transition Lane主线程繁忙时允许它等待或让位

Lane 决定“这轮 Render 包含哪些 update”；Scheduler 优先级决定“这轮 Render 的回调何时执行”。映射后，Scheduler 不再关心具体 Lane，也不读取 Fiber。

MessageChannel：暂停后，通知自己继续工作

React 的一次 Render 可能很长。如果一直占着主线程，浏览器就不能及时处理点击和绘制页面。因此 React 计算一小段后会先停下来，把主线程还给浏览器。

React 计算一小段 → 暂停 → 浏览器处理点击、绘制页面 → React 继续计算下一段

MessageChannel 只负责最后一步：安排一次稍后执行的回调，让 React 可以继续未完成的工作。

// 简化理解，不是完整源码
port1.onmessage = continueWork;  // 收到消息后，继续计算

if (这一小段时间用完了) {
  port2.postMessage(null);   // 通知：稍后继续
  return;                    // 现在先把主线程还给浏览器
}
一句话记忆：MessageChannel 不负责拆 Fiber，也不会让计算变快；它只是让“继续计算”排到稍后，使浏览器有机会在两段计算之间处理更紧急的事情。

这里使用 MessageChannel，是因为它能较快地安排下一次回调，又不像连续嵌套的 setTimeout 那样可能受到最小延迟限制。时间片是否结束，则由 Scheduler 的 shouldYield() 判断。

🔄 一轮的节奏(不停循环,直到没活):
① postMessage 排一个宏任务 → ② 浏览器:绘制 / 响应输入 → ③ onmessage 触发,干够 5ms → 还有活? 是 → 回到 ① ↺

每轮工作结束后，浏览器可以处理绘制和输入事件，这构成了时间切片的基础。

// ① 建通道:port1 收到消息就执行 performWorkUntilDeadline
const channel = new MessageChannel();
const port = channel.port2;
channel.port1.onmessage = performWorkUntilDeadline;

// ② 安排工作 = 让 port2 发一条消息 → 排进「宏任务」队列(不是立刻执行)
function requestHostCallback(cb){
  scheduledHostCallback = cb;
  port.postMessage(null);      // ★发消息 = 排一个宏任务
}

// ③ 每个宏任务里:干一片活;还有活就再发消息,排下一片
const performWorkUntilDeadline = () => {
  startTime = getCurrentTime();                          // 记本片起点(shouldYield 要用)
  const hasMore = scheduledHostCallback(true, startTime); // 干活(内部每个单元后看 shouldYield)
  if(hasMore) port.postMessage(null);              // 还有活 → 排下一个宏任务,续下一片
};

// 「这一片 5ms 到了没?」——工作循环每处理一个单元就问它一次
const frameInterval = 5;
function shouldYieldToHost(){
  return getCurrentTime() - startTime >= frameInterval;
}

6 · Lane：选择本次更新

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
这一章建立 Lane 的完整工作模型

Lane 不只是“高、中、低”标签。它把每个 update 的优先级编码成二进制位，让 React 能在多种更新同时存在时，选择一批工作、跳过暂时不做的更新，并在之后按原顺序补算。

阅读顺序：为什么需要 → Lane/Lanes 是什么 → 如何分配与传播 → root 如何选一批 Lanes → Render 如何处理 → 挂起、唤醒、过期和提交后怎样变化。

先回答：React 为什么需要 Lane

同一个 root 上可以同时存在不同来源、不同紧急程度的更新。例如用户正在输入文字，页面还要重新筛选一个大列表：

输入框更新用户正在等待反馈，应尽快显示。
列表筛选更新计算量大，可以放进 startTransition 延后处理。

如果 React 只用普通 FIFO 队列，就只有两个不理想的选择：严格按顺序执行，可能让大列表阻塞输入；或者粗暴地把低优更新挪走，又容易破坏同一 state 队列的计算顺序。Lane 的作用是：保留 update 的原始队列顺序，同时额外标记“这一轮要不要处理它”。

核心模型：UpdateQueue 保存“更新按什么顺序发生”；Lane 决定“本轮选择其中哪些更新”；baseQueue 保存“本轮跳过、以后必须按原顺序重放的更新”。三者配合，React 才能既优先响应紧急工作，又不算错最终 state。

Lane、Lanes 与 renderLanes 分别是什么

名称含义典型位置
Lane一个二进制位，通常表示一个 update 所属的优先级通道update.lane
Lanes多个 Lane 按位 OR 得到的集合，不是一种新的优先级fiber.lanes、root.pendingLanes
renderLanes当前 Render 选择处理的 Lane 集合，可能只有一个 bit，也可能包含多个Render 工作循环与 update 队列处理函数的参数
Lane 不是线程、任务或 state。它本质上是数字中的一个 bit。React 用位运算快速完成“加入集合、判断是否包含、合并、移除和寻找最高优先级组”。

第一步：创建 update 时分配 Lane

调用 setState 后，React 创建 update，并通过 requestUpdateLane() 根据当前上下文决定它属于哪类 Lane。常见来源如下：

更新来源常见 Lane含义

第二步：Lane 从更新节点传播到 root

只有 Button 知道自己更新还不够，负责调度的是根节点。React 必须沿 Fiber 的 return 指针向父级走，让 root 知道“Button 这条分支有一个 Sync 更新”。源码把这个向上传递过程叫作冒泡。

三个字段不要混：Button.lanes 表示“我自己有更新”；Main/App.childLanes 表示“我的后代有更新”；root.pendingLanes 表示“整棵应用还有哪些 Lane 没处理”。

为什么祖先要记录 childLanes？Render 从 root 往下走时，可以据此找到 Button 所在的分支。某个分支的 childLanes 不包含本次 Lane，就说明下面没有相关更新，整棵子树可以跳过。

root 不只保存 pendingLanes

更新到达 root 后，会进入多个 Lane 集合中的一种状态。理解这些字段，才能看懂 getNextLanes 为什么有时选择、有时跳过某些工作：

root 字段表示什么什么时候变化
pendingLanes已经到达 root、但还没有全部完成的 Lane新 update 到来时加入；成功提交并确认完成后移除
suspendedLanes本次尝试时因数据等原因挂起的 LaneRender 遇到 Suspense 后标记
pingedLanes之前挂起，但依赖的数据已经就绪，可以重试的 LanePromise resolve 后由 ping 标记
expiredLanes等待太久、需要防止继续饿死的 Lane超过过期时间后标记，后续提高完成紧迫性
entangledLanes与其他 Lane 有共同完成关系的 Lane相关更新不应独立提交中间状态时建立关联

第三步：getNextLanes 选择下一批工作

getNextLanes(root) 的目标不是“永远取最高的一个 bit”，而是从 root 的各种 Lane 状态中选出下一批值得执行、能够执行，并且适合一起执行的工作。可以按下面的顺序理解：

  1. 没有 pending 工作：返回 NoLanes。
  2. 先考虑非 Idle 工作：普通交互和页面更新完成前，不急着处理 Idle。
  3. 优先选择未挂起的 Lane：从 pendingLanes 中排除仍在 suspended 且尚未 ping 的部分。
  4. 没有新鲜可执行工作时，再看 pinged：数据已经就绪的挂起任务可以重试。
  5. 从候选中取最高优先级组：得到的可能是一组 Lane，而不一定只有一个 bit。
  6. 与正在进行的 WIP 比较：新 Lane 没有更高优先级时，React 通常继续已有工作，避免白白丢掉进度。
  7. 计算共同完成关系：进入 Render 时再纳入需要一起处理的 entangled Lane。
“选最高优先级”不等于“只选一个 Lane”。React 先用最高 bit 判断当前属于哪个优先级组，再返回这个组中适合共同处理的 Lanes；之后还可能考虑 entanglement 或特殊同步刷新。

假设大列表的 Transition 仍在等待，此时用户又输入一个字符，root 变成：

这里分开不是因为一次 Render 技术上只能处理一个 Lane，而是一次调度决策：假设 Sync 只需 2ms，Transition 的大列表需要 80ms；若总是合并，用户可能要等完整计算结束才看到输入结果。若两个 Lane 因共同完成关系被 entangle，或进入特殊强制刷新路径，React 也能够让它们处于同一批有效 Render 范围。

第四步：Render 用 Lanes 筛选 update 与子树

Render 遍历 update 队列时，不是比较“它是不是唯一被选中的 Lane”，而是检查 update.lane 是否属于 renderLanes 这个集合：

if (isSubsetOfLanes(renderLanes, update.lane)) {
  state = applyUpdate(state, update); // 本轮包含它：执行
} else {
  cloneToBaseQueue(update);          // 本轮不包含：保留以后重放
}
因此一次处理多个 Lane 时不需要另一套算法：renderLanes 中有几个 bit，就能匹配几类 update。Sync 与 Transition 若都在集合中，两类 update 都通过检查；若集合里只有 Sync，Transition 就被跳过并保留。

Lane 还用于跳过整棵子树。若某个 Fiber 自己的 lanes 和后代的 childLanes 都不与本轮有效 Lanes 相交，beginWork 可以直接 bailout，不再执行该分支的组件和 Diff。

Lane 只负责表达“这轮包含哪些更新”以及它们的优先级关系。Lane 本身不会暂停 JavaScript，也不会保存或丢弃 WIP；取消旧任务、决定续算还是重算，属于下一章 Scheduler 与 Fiber 工作循环的配合。

二进制位到底怎样表示一组 Lane

先只用 4 位演示。每一种 Lane 独占一位；1 表示“集合中有这种 Lane”，0 表示“没有”。

Lane二进制读法
SyncLane0001第 1 位是 1
DefaultLane0100第 3 位是 1
TransitionLane1000第 4 位是 1

操作一：用 OR 把 Lane 加进 pendingLanes

root 已经等待一个 Transition，又来了一个 Sync。逐位执行 OR：任意一边是 1，结果就是 1。

  1000   // Transition
| 0001   // Sync
------
  1001   // pendingLanes：两种更新都还没处理

因此 root.pendingLanes = 1001 不是一种新优先级，而是一个集合：里面同时有 Sync 和 Transition。

操作二：用 AND 判断本轮是否包含某个 update

假设本轮只选择 Sync，所以 renderLanes = 0001：

  0001   // renderLanes
& 0001   // Sync update.lane
------
  0001   // 非 0：本轮处理 Sync update

  0001   // renderLanes
& 1000   // Transition update.lane
------
  0000   // 等于 0：本轮跳过 Transition update

操作三：从集合中取出最高优先级

React 把更低的二进制位定义为更高优先级，因此 getHighestPriorityLane(1001) 会取出最右侧的 1，得到 0001，也就是 Sync。源码常见写法是 lanes & -lanes；现阶段只需记住它的结果是“保留最右侧的 1”。

const nextLane = pendingLanes & -pendingLanes;
// pendingLanes = 1001 → nextLane = 0001 → SyncLane
三个运算对应三个问题：| 把 Lane 加入集合；& 判断集合是否包含某 Lane；lanes & -lanes 取出最高优先级 Lane。真实源码位数更多，但读法完全相同。

14 · 并发渲染 Concurrent Mode

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

并发 = 可中断、可恢复、可抢占;能被中断的是 workLoop 这个 while 循环,不是任何函数。

本章包括：① workLoop 可中断原理 ② 为何只有 Render 可中断 ③ 中断后丢弃 WIP ④ 靠 baseQueue 恢复。

并发渲染建立在 Render 阶段可以暂停的基础上。这里需要区分两种情况：

停法①:时间片让位(暂停,不丢)

每片约 5ms,shouldYield() 为真就 break。这不是丢弃!workInProgress 指针原地保留,performConcurrentWorkOnRoot 返回「自己」作为续作,下一个时间片从原地接着跑。同一次渲染,只是切成了好几片。

停法②:高优抢占(丢弃,重来)

渲染低优期间,来了更高优的更新。ensureRootIsScheduled 发现后取消当前任务、重排高优。这次才丢弃半成品 WIP,从 current 重建、先渲染高优。低优之后再从头来过。

function workLoopConcurrent(){
  while(workInProgress !== null && !shouldYield()){ // ★每处理完一个单元就检查
    performUnitOfWork(workInProgress);            // 处理一个 fiber,瞬间完成
  }                                                // 用完 5ms 就 break(指针保留)
}

渲染入口:renderRootConcurrent + prepareFreshStack

上面两种「停」的区别,全在这段入口逻辑里:

function performConcurrentWorkOnRoot(root){
  const lanes = getNextLanes(root);
  const status = renderRootConcurrent(root, lanes);
  if(status === RootInProgress){        // 被时间片打断、还没渲完
    return performConcurrentWorkOnRoot.bind(null, root); // ★返回自己=续作,下片继续
  }
  commitRoot(root);                      // 渲完了才提交
}
function renderRootConcurrent(root, lanes){
  if(workInProgressRoot !== root || workInProgressRootRenderLanes !== lanes){
    prepareFreshStack(root, lanes);   // 优先级变了 → 以 current 建全新 WIP(丢弃旧的)
  }                                     // 优先级没变(时间片续作)→ 不重建,接着旧 WIP 跑
  workLoopConcurrent();
}

关键:时间片续作时 lanes 没变 → 跳过 prepareFreshStack,接着旧 WIP;抢占时 lanes 变了 → prepareFreshStack 用 createWorkInProgress 从 current 重建根 WIP,旧的半成品被丢弃。

为什么只有 Render 阶段可中断

Render 是纯内存计算、无副作用——WIP 只是内存对象,丢了重来没代价,用户也看不见(屏幕上一直是完好的 current 树)。Commit 直接改真实 DOM + 触发生命周期,有外部副作用,中途停会画面撕裂、副作用重复,所以必须同步、不可断。

抢占的完整时间线(看一遍就懂)

场景:正在低优渲染一个大列表,渲到一半你点了个按钮(高优)。

低优·片1低优·片2 ⚡ 你点击(高优 setState) 丢弃低优 WIP→ 高优·片1commit 高优 ✨→ 重新渲低优·片1片2commit 低优 ✨

高优插队 → 丢弃低优半成品 → 先把高优渲染并上屏(你的点击立刻有反应)→ 再从头重渲低优。全程屏幕上都是完好画面,你感觉「点哪都跟手」。

丢了低优 WIP,那它算了一半的 state 呢? 靠 Trigger·UpdateQueue 的 baseQueue:被跳过/未消费的 update 都存在 baseQueue 里,重跑时从 baseState + baseQueue 重新 reduce,结果依然正确。这就是"可恢复"的底层保证。

互动:递归无法中断 vs 循环可中断+恢复

怎么操作：点击开始，处理过程中立即点击“插入高优任务”。左侧观察调用栈为何不能保存中间位置，右侧观察 workInProgress 指针如何暂停并恢复。

循环:workInProgress 指针(可存档)

15 · 两大并发 API 底层

useTransition 把「更新」降优先级,useDeferredValue 把「值」降优先级——本质都是让高优交互能抢占。

本章包括：两者原理、区别、使用场景、底层 Lane 差异。

useTransition

把更新标记为低优。startTransition(fn) 内的 setState 拿到 TransitionLane,可被用户输入等高优打断。返回 isPending 标识过渡进行中。

const [isPending, startTransition] = useTransition();
startTransition(() => {
  setList(bigFilteredList);  // → TransitionLane(低优)
});

useDeferredValue

把值降级。返回一个"滞后版"的值,React 先用旧值渲染(高优),再在低优 lane 用新值渲染一遍。适合你拿不到 setState(值来自 props)的场景。

const deferredQuery = useDeferredValue(query);
// 输入框用 query 即时更新,列表用 deferredQuery 滞后更新

怎么用:两个真实场景(照着抄)

场景 A · 搜索框打字卡顿 → useDeferredValue

受控输入 + 一个昂贵的派生视图(过滤大列表 / Markdown 预览 / 图表)。你拿不到列表的 setState(它是 query 派生的),就用 useDeferredValue 包住那个「值」:

function SearchPage(){
  const [query, setQuery] = useState('');
  const deferredQuery = useDeferredValue(query); // 滞后版的 query
  return (
    <>
      <input value={query} onChange={e => setQuery(e.target.value)} />  // 输入框用 query:即时、跟手
      <BigList query={deferredQuery} />   // 大列表用 deferredQuery:稍后跟上,不卡输入
    </>
  );
}

效果:你飞快打字时,输入框每个字都即时显示(高优);而重型列表用「滞后值」在空档里慢慢追上,不会因为列表重渲染卡住你打字。

场景 B · 切换重型 tab / 提交搜索 → useTransition

一次交互会触发昂贵渲染,但你想让交互本身保持跟手。你能拿到那段 setState,就用 startTransition 把它标成「非紧急」;isPending 还能显示 loading:

function Tabs(){
  const [tab, setTab] = useState('home');
  const [isPending, startTransition] = useTransition();
  function select(next){
    startTransition(() => setTab(next)); // 这次切换是「可被打断的低优」
  }
  return (
    <>
      <button onClick={() => select('posts')}>Posts</button>
      {isPending && <Spinner/>}        // 过渡中显示 loading
      <TabContent tab={tab}/>            // 切到很重的 tab 时,旧内容先留着,不白屏
    </>
  );
}

效果:点按钮瞬间响应,重型 tab 内容在后台低优渲染(可被你下一次点击打断),ready 后再换上;期间用 isPending 显示转圈。

选择方式：你能改那段 setState → 用 useTransition(还能拿 isPending 显 loading);你手里只有一个值(通常来自 props/上游),改不到它的 setState → 用 useDeferredValue。
📋 常见适用场景
  • 大列表实时过滤 / 搜索(输入跟手、结果稍后)
  • 切换重型 tab / 路由(旧页先留,新页后台渲染)
  • 输入时的实时预览:Markdown、代码高亮、图表重绘
  • 自动补全下拉:输入即时、候选列表滞后

⚠️ 不要滥用:只在「某段渲染确实重、且会拖慢交互」时用。轻量更新加了反而多一次渲染。

17 · 批量更新原理

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

同一批多次 setState 只渲染一次 = 批量更新。

本章包括：① 三种批量(16/17 同步、异步漏网、18 自动)② Lane 如何实现自动合并。

三种批量更新

时期行为
React 16/17 同步批量只在 React 事件回调内批处理(靠 batchedUpdates + executionContext 标志)
React 16/17 异步"漏网"setTimeout/Promise/原生事件里的多次 setState 不批处理,各触发一次渲染

① React 16/17:靠「执行上下文标志」批处理(所以会漏网)

16/17 里,React 给自己的事件回调套了一层 batchedUpdates:进回调前把全局标志 executionContext 打上「批处理中」,回调里的 setState 就只入队、不刷新;回调结束再统一 flush 一次。

function batchedUpdates(fn){
  const prev = executionContext;
  executionContext |= BatchedContext;   // 打上「批处理中」标志
  try { return fn(); }             // 你的事件回调:多次 setState 只入队
  finally {
    executionContext = prev;            // 回调结束,清标志
    if(executionContext === NoContext) flushSyncCallbacks(); // 统一刷一次
  }
}
为什么 setTimeout/Promise 里会「漏网」:这些回调是浏览器/微任务队列直接调的,不经过 React 的 batchedUpdates 包裹 → executionContext 没打标志 → 每个 setState 都当场各刷一次渲染。

② React 18:靠「微任务 + 去重」批处理(所以哪都能批)

18 不再依赖执行上下文标志。每次 setState → scheduleUpdateOnFiber → ensureRootIsScheduled。对 SyncLane,它用 微任务安排 flush,而且会去重:

function ensureRootIsScheduled(root){
  const newPriority = getHighestPriorityLane(getNextLanes(root));
  if(root.callbackPriority === newPriority) return; // ★同优先级已排过 → 直接返回(去重!)
  if(newPriority === SyncLane) scheduleMicrotask(flushSyncCallbacks); // 安排一个微任务
  else scheduleCallback(…);
  root.callbackPriority = newPriority;
}

因为 flush 挂在「微任务 / 本轮事件循环末尾」,而不是「React 回调末尾」,所以 setTimeout、Promise、原生事件里的多次 setState 也落在同一批、只刷一次——这就是「任何地方都自动批处理」。

③ 一步步看:Promise 里连写 3 次 setState

④ flushSync:主动跳出批处理

flushSync(() => setCount(1));  // 立即同步渲染,不等微任务
// 下一行就能读到更新后的 DOM(如需立刻测量布局)

flushSync 内部临时设好执行上下文,并在结束时立刻 flushSyncCallbacks,不等微任务。代价是丢掉批处理的性能收益,只在确实需要「同步拿到结果」时用。

⑤ 并发(Transition)下的批量

startTransition 里的多次 setState 落在同一个 TransitionLane,同样只渲染一次。但如果一批里混了不同优先级(比如一个 Sync、一个 Transition),会分成两批各自渲染——高优先渲并上屏,低优随后。这也是靠 Lane 区分的。

🗺 本章路线图

子组件没准备好就 throw 一个 Promise,最近的 Suspense 边界捕获它、先渲染 fallback,Promise resolve 后重渲染。

本章包括：挂起机制、与 Scheduler 协同、优先级降级。

挂起机制:靠 throw 一个 Promise

当子组件的数据/代码没就绪,它在渲染时 throw 一个 Promise(不是错误)。最近的 Suspense 边界捕获它:

  • Reconciler 捕获到抛出的是 Promise → 把该 Suspense fiber 标记为"挂起",渲染其 fallback 而非 children。
  • 给 Promise 挂上 .then:resolve 后触发一次重新渲染,这次 children 能正常渲染,替换掉 fallback。
  • React.lazy 就是把动态 import() 包装成"未加载时 throw Promise"的组件,天然配合 Suspense。
// 概念示意:数据未就绪就抛 Promise
function read(){
  if(status==='pending') throw promise;   // ★被 Suspense 捕获
  if(status==='resolved') return data;
}

与 Scheduler 协同 & 优先级降级

配合并发,Suspense 可以做到"避免 loading 闪烁":高优更新若会导致已显示内容退回 fallback,React 可用 transition 优先级让旧 UI 继续显示、在后台低优渲染新内容,ready 后再切换。边界的 fallback 渲染本身也是一次正常的 Reconcile,受同一套 Lane 调度。

本章小结：「throw Promise」挂起 → 渲染 fallback → resolve 后重渲染换回内容;lazy 就是把 import() 包成会 throw 的组件;配合并发可避免 loading 闪烁。→ 下一章:第五层·合成事件。

19 · 合成事件系统

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

React 把事件统一委托到根容器,靠 __reactFiber 反查 fiber、沿 fiber 树模拟捕获/冒泡,派发归一化的 SyntheticEvent。

本章包括：事件委托、事件池、合成 vs 原生顺序、stopPropagation 为何有时无效。

事件委托

React 不给每个元素绑监听器,而是把所有事件统一委托到根容器(React 17+ 是 createRoot 的 container,16 是 document)。DOM 节点上挂 __reactFiber$xxx 指回 fiber,事件触发时反查目标 fiber,再沿 fiber 树的 return 链模拟捕获/冒泡,派发归一化的 SyntheticEvent。

事件池(event pooling)

React 16 会复用 SyntheticEvent 对象(用完清空属性),异步访问 e.target 会拿到 null,需 e.persist()。React 17 起移除了事件池,可放心异步使用事件对象。

合成事件 vs 原生事件顺序 / 为什么 stopPropagation 有时无效

  • 执行顺序:原生事件先在真实 DOM 冒泡,到达委托根后 React 才开始派发合成事件。所以子元素的原生监听器早于父元素的 React 合成监听器触发。
  • 为什么阻止冒泡有时无效:合成事件的 e.stopPropagation() 只能阻断合成事件体系内的冒泡,拦不住已经发生的原生冒泡;反之在原生监听器里 stopPropagation 若阻止了事件到达委托根,会导致整个 React 合成事件不触发。混用原生 + 合成事件是这类 bug 的根源。

互动:点击叶子,沿 fiber 树捕获→冒泡

怎么操作：先直接点击最内层按钮，观察捕获阶段从外到内、冒泡阶段从内到外；重置后勾选 stopPropagation，查看派发在哪一层停止。
点击按钮开始派发。
本章小结：事件委托到 root、按 fiber 树的 return 链收集监听器模拟冒泡;17 起移除事件池;原生事件先于合成事件;混用原生+合成时 stopPropagation 易失效。→ 下一章:第六层·渲染优化。

20 · 渲染优化底层

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

本节内容：识别可以触发 bailout 的条件，减少不必要的组件执行和 DOM 操作。

本章包括：① bailout 触发条件 ② memo 浅比较底层 ③ 怎么做到精准局部更新。

bailout:满足条件就不渲染

beginWork 里,若同时满足以下条件,React 走 bailoutOnAlreadyFinishedWork,跳过该 fiber 的重渲染,直接复用旧子树:

  • oldProps === newProps(引用相等)且 context 未变;
  • 该 fiber 没有自己的待处理更新(!includesSomeLane(renderLanes, fiber.lanes));
  • 若连 childLanes 也没有待更新 → 整棵子树都跳过(这就是精准局部更新的关键:childLanes 让 React 知道子树里哪还有活)。

memo 的浅比较

function shallowEqual(objA, objB){
  if(is(objA, objB)) return true;
  const keysA=keys(objA), keysB=keys(objB);
  if(keysA.length !== keysB.length) return false;
  for(const k of keysA)              // 只比第一层
    if(!is(objA[k], objB[k])) return false;
  return true;
}

只比 props 第一层。所以传新建的对象/数组/函数(每次渲染引用都变)会让 memo 失效 —— 这正是要配 useMemo/useCallback 稳定引用的原因。深层嵌套对象浅比较也测不出,需自定义 areEqual。

如何做到精准局部更新

state 定义得越靠近使用它的组件,重渲染范围越小(状态下放)。配合 childLanes 的 bailout,只有"从 root 到发生更新的 fiber"这条路径 + 其必要子树会被 beginWork,兄弟子树整体跳过。

本章小结：bailout = props 引用没变 + 无自身更新 + context 没变 → 复用旧子树,childLanes 为 0 还能整棵跳过;memo 只浅比较第一层,传新建对象/函数会失效,要配 useMemo/useCallback。→ 下一章:第六层·渲染 bug。

21 · 常见渲染 bug 的底层解释

🧭 主线setState→Update→Lane→Scheduler→Render→Effect→Commit→DOM
🗺 本章路线图

本节内容：从更新队列、闭包、key 和批处理机制解释四种常见问题。

本章包括：setState 为何时同步时异步、多次 setState 为何合并、useTransition 为何能解卡顿、useEffect 依赖写错为何死循环。

为什么 setState 有时同步、有时异步?

setState 本身从不真正同步更新 state——它总是异步批处理。所谓"同步"的错觉:在 React 16/17 的事件回调里,setState 后立即读 this.state/state 变量拿到的是旧值(还没渲染);而在 setTimeout 里(16/17 不批处理)每次 setState 会各自触发一次渲染,看起来"立即生效"。React 18 全局自动批处理后行为统一。要立即拿新值:flushSync 或 useReducer/函数式更新。

为什么多次 setState 会合并?

它们的 update 都进了同一个 updateQueue.shared.pending 环状链表,ensureRootIsScheduled 去重只排一次渲染,渲染时一起 reduce。setCount(count+1) 连写三次因闭包里 count 是旧值只 +1;用函数式 setCount(c=>c+1) 才会累加,因为每个 update 拿到的是前一个的结果。

为什么 useTransition 能解决卡顿?

它把大列表渲染降到 TransitionLane(低优),于是这段渲染在 Concurrent 下可被时间切片、可被用户输入抢占。输入这类高优更新随时打断大渲染先响应,页面就跟手不卡(详见第四层·Concurrent/第四层·并发API)。

为什么 useEffect 依赖写错会无限循环?

典型:effect 里 setState,但依赖数组包含了一个每次渲染都新建的引用(对象/数组/函数),或该 state 本身。流程:渲染 → effect 执行 → setState → 重渲染 → 依赖"变了" → effect 再执行 → …死循环。修法:用 useCallback/useMemo 稳定依赖、函数式更新去掉对 state 的依赖、或把不该进依赖的挪出去。

Trigger 阶段完成：setState 产生 update，并将它放入更新队列。下一步进入 Schedule：由 Lane 标记和选择更新优先级。

22 · SSR / Hydration / 流式渲染

🗺 本章路线图

SSR 在服务器先生成 HTML(首屏快),再由客户端 hydration「激活」。

本章包括：hydration 是什么、mismatch 怎么来、React 18 流式渲染 + 选择性注水。

客户端渲染(CSR)首屏是一张空 HTML,JS 下载执行后才有内容 —— 首屏慢、SEO 差。服务端渲染(SSR)在服务器先把组件渲染成 HTML 字符串发给浏览器,首屏立刻可见,再由客户端「激活」。

Hydration(注水/激活)

服务器发来的 HTML 是「死」的(没有事件、没有 state)。Hydration 是客户端 React 复用这些已有 DOM、给它们绑定事件、挂上 Fiber/state,让它「活」过来的过程。关键:不重新创建 DOM,只是附着。

// 客户端:hydrateRoot 而非 createRoot
import { hydrateRoot } from 'react-dom/client';
hydrateRoot(document.getElementById('root'), <App/>);
// 服务端
import { renderToPipeableStream } from 'react-dom/server';
Hydration 不匹配(mismatch):若服务端渲染的 HTML 与客户端首次渲染结果不一致(如用了 Date.now()、window、随机数),React 会警告并可能丢弃服务端 HTML 重新渲染,首屏优势尽失。用 useEffect 里读浏览器专属值、或 useId 保证 id 一致来避免。

React 18 两大 SSR 升级:流式 + 选择性 Hydration

  • Streaming SSR(流式渲染):renderToPipeableStream 边渲染边把 HTML 分块发给浏览器,不必等整页渲染完。慢的部分(数据未就绪)用 <Suspense> 包住先发 fallback,数据好了再流式补发那一块 HTML + 内联脚本替换。
  • Selective Hydration(选择性注水):不必等整棵树的 JS 都加载完才能交互。React 按 Suspense 边界分块注水,而且用户点击哪块就优先注水哪块(利用并发的 Lane 优先级)。这就是 SSR 与并发架构的结合点。

HTML 分块流式传输,浏览器逐块显示并按需注水

本章小结：hydration = 复用服务端 HTML、绑事件挂 state 让它「活」;18 支持流式(边渲染边发 HTML)+ 选择性注水(点哪块先激活哪块);mismatch 来自服务端/客户端首渲不一致。→ 下一章:第七层·Server Components。

23 · React Server Components(RSC)

🗺 本章路线图

RSC 让一部分组件只在服务器运行、代码不进 bundle,只有交互的「叶子」下沉到客户端。

本章包括：Server vs Client 组件、'use client'/'use server'、RSC Payload。

RSC 是比 SSR 更进一步的架构。SSR 是「在服务器跑一遍组件生成 HTML,然后整个应用还要在客户端再 hydrate」;RSC 则让一部分组件只在服务器运行、永不发送到客户端,连它们的代码都不进 bundle。

Server Component vs Client Component

Server Component(默认)Client Component('use client')
运行位置只在服务器服务器预渲染 + 客户端 hydrate
能否用 state/effect/事件不能(无 useState/useEffect/onClick)能
能否直接读数据库/文件能(async 组件里直接 await)不能
代码进 bundle 吗不进,减小体积进
// 服务器组件:可以是 async,直接取数据,零客户端 JS
async function ProductPage({id}){
  const product = await db.products.find(id);  // 直接查库
  return <div>
    <h1>{product.name}</h1>
    <AddToCartButton id={id}/>   // ← 这个需交互,是客户端组件
  </div>;
}
// AddToCartButton.jsx
'use client';   // ★声明这是客户端组件,才能用 onClick/useState

底层:RSC Payload 与边界

服务器不把 Server Component 渲染成 HTML,而是渲染成一种特殊的序列化格式(RSC Payload) —— 一棵描述 UI 的树,其中客户端组件是「占位引用」(指向要加载的 JS chunk)。客户端拿到 payload 后拼装,只为标了 'use client' 的部分下载和 hydrate JS。'use server' 则用于 Server Actions(客户端能直接调用的服务器函数)。

RSC 通常由框架(Next.js App Router 等)集成,不是裸 React 直接用。理解要点:组件默认在服务器,交互的「叶子」才下沉到客户端,从而大幅减少发到浏览器的 JS。
本章小结：默认服务器组件(能 await 取数据、零客户端 JS),标 'use client' 的才下载/hydrate;服务器把树序列化成 RSC Payload,客户端拼装。→ 下一章:第七层·React 19。

25 · 外部状态与「撕裂」(tearing)

🗺 本章路线图

并发渲染下,不同组件可能读到同一外部状态的不同版本 = 撕裂(tearing)。

本章包括：撕裂为什么发生、useSyncExternalStore 怎么解决。

Redux、Zustand、以及任何存在 React 之外的状态源,在并发渲染下会遇到一个隐蔽问题:撕裂(tearing) —— 同一次渲染中,不同组件读到了同一个外部状态的不同版本,导致 UI 不一致。

撕裂为什么发生

并发渲染是可中断的。假设组件 A、B 都读外部 store:渲染 A 时 store=1,渲染被时间切片打断,期间外部把 store 改成 2,恢复后渲染 B 读到 2 → 同一帧里 A 显示 1、B 显示 2。同步渲染不会有这问题(一口气读完),并发才暴露。

解法:useSyncExternalStore

React 18 提供 useSyncExternalStore，用于在并发渲染中订阅外部 store。React 会比较快照；如果渲染期间快照发生变化，则重新检查并以同步方式完成更新，避免同一次提交中的组件读取到不同版本。Redux 等状态库可以通过这一接口与 React 集成。

const state = useSyncExternalStore(
  store.subscribe,      // 订阅:变化时通知 React 重渲染
  store.getSnapshot,    // 读当前值(React 会用它校验一致性)
  store.getServerSnapshot // SSR 时的值(可选)
);
问题来源：渲染可以中断，而外部数据可能在中断期间发生变化。useSyncExternalStore 提供统一的订阅和快照读取方式，使 React 能检测变化并保持不同组件读取结果的一致性。
本章小结：并发可中断 → 渲染途中外部数据变了 → 同一帧不同组件读到不同值 = 撕裂;useSyncExternalStore 检测到就退回同步渲染保证一致;现代状态库内部都用它。→ 下一章:第七层·React Compiler。

26 · React Compiler(自动记忆化)

🗺 本章路线图

React Compiler 在编译期自动插入记忆化,让你几乎不用手写 memo/useMemo/useCallback。

本章包括：它做什么、为什么依赖「组件是纯的」、对手动 memo 的影响。

手动 useMemo/useCallback/memo 心智负担重、容易写错或漏写。React Compiler(原 React Forget)是一个编译期工具,自动分析组件、在构建时插入等价的记忆化代码,让你几乎不再需要手写 memo 系列。

它做什么

  • 静态分析每个组件的渲染逻辑,识别哪些值依赖哪些输入。
  • 自动缓存组件内的计算结果与子元素,只有真正依赖变化时才重算/重渲染 —— 相当于自动、细粒度的 useMemo。
  • 产出仍是标准 React 代码,运行时不变,只是「记忆化」被编译器包办。
// 你写(无任何 memo):
function TodoList({todos, filter}){
  const visible = todos.filter(t => match(t, filter));
  return <List items={visible}/>;
}
// 编译器自动产出 ≈ 用缓存包住 visible 和 <List>,
// 仅当 todos 或 filter 变化时才重算,等价于全自动 useMemo。

前提:遵守 React 规则

编译器能安全优化的前提是你的组件是「纯」的、遵守 Hooks 规则、不在渲染中产生副作用。这也是为什么 StrictMode 双调用、ESLint 规则如此重要 —— 它们帮你写出编译器能优化的代码。它是渐进采用的:可整项目、也可按文件开启。

趋势判断:未来 useMemo/useCallback 会从「性能必备」退化为「少数手动兜底」。但理解它们的原理依然重要 —— 你得知道编译器在替你做什么、什么情况下它会放弃优化。
本章小结：编译器静态分析、自动缓存计算与子元素 = 全自动 useMemo;前提是组件纯、守 Hooks 规则(所以 StrictMode/ESLint 重要);未来手写 memo 退为兜底。→ 下一章:第七层·高级模式与陷阱。

27 · 高级模式与常见陷阱

🗺 本章路线图

几个高级 API 与它们的常见陷阱,直接影响项目质量。

本章包括：Error Boundary、Portal、forwardRef/useImperativeHandle、StrictMode 双调用、Context 性能陷阱、useId。

Error Boundary(错误边界)

类组件通过 getDerivedStateFromError / componentDidCatch 捕获子树渲染错误,展示兜底 UI(函数组件目前无等价 Hook,需用类或库)。注意:它捕获不到事件处理器、异步代码、SSR 里的错误。

Portal 与 forwardRef / useImperativeHandle

  • createPortal(child, domNode):把子节点渲染到父 DOM 层级之外(如挂到 body),但事件仍沿 React 树冒泡(合成事件按 fiber 树,不按真实 DOM)。适合弹窗、tooltip。
  • useImperativeHandle:配合 ref 暴露受控的命令式接口(如 inputRef.current.focus()),而不是整个 DOM。React 19 起 ref 可直接作 prop,forwardRef 不再必需。

StrictMode 双调用

开发模式下 <StrictMode> 会故意双调用渲染函数、以及 effect 的「挂载→卸载→再挂载」。目的:暴露不纯的渲染和缺失的 effect 清理。如果你的组件双调用后出 bug,说明它有副作用不纯 —— 也正是 React Compiler 能否优化它的分界线。生产环境不会双调用。

Context 性能陷阱

Context value 一变,所有消费者都重渲染(绕过 memo)。陷阱:把频繁变化的值和稳定的值放同一个 Provider,或每次渲染传新对象字面量作 value。优化:拆分 Context(变与不变分开)、value 用 useMemo 稳定、或用 useSyncExternalStore/状态库做选择性订阅。

useId

useId() 生成服务端/客户端一致的唯一 id,专为解决 SSR hydration 中 id 不匹配(不能用随机数/递增计数)。用于 aria-*、label/input 关联等。

C1 · Context 源码原理

🗺 本章路线图

很多人会用 useContext,却说不清「为什么 Provider 一变,深层组件即使被 memo 包住也更新」。

本章包括：值栈 push/pop、定向传播(为何穿透 memo)。

很多人只会用 useContext,但说不清「为什么 Provider 一变,深层组件即使被 memo 包住也会更新」。这背后是值栈 + 定向传播两套机制。

createContext 与值栈(stack)

createContext(default) 返回一个对象,里面有 Provider、Consumer 和一个 _currentValue。渲染遇到 <Provider value=x> 时,reconciler 在 beginWork 把旧值压栈、写入新值,在 completeWork 弹栈还原 —— 这样嵌套的同一个 Context 的多个 Provider 才能各自作用在自己的子树范围内。

// beginWork 进入 Provider(pushProvider 简化)
push(valueStack, context._currentValue);  // 旧值压栈
context._currentValue = newValue;         // 写入本层新值
// completeWork 离开 Provider(popProvider)
context._currentValue = pop(valueStack);   // 还原为上层的值

定向传播:为什么能绕过 memo / bailout

组件调用 useContext(ctx) 时,除了读值,还会把当前 fiber 登记到该 context 的依赖上。当 Provider 的 value 变化(Object.is 判不等),reconciler 执行 propagateContextChange:从 Provider 向下深度遍历子树,凡是登记了该 context 的 fiber,直接给它标记一个更新 lane —— 即使这条路径上有组件本可以 bailout(memo/props 没变),也会因为「有 context 依赖的待更新」而被强制往下走到消费者。这就是「Context 变化穿透 memo」的源码真相。

性能启示(呼应第七层·高级模式与陷阱):正因为是「Provider 一变→所有消费者全更新」,才要拆分 Context(把频繁变和不常变的分开)、用 useMemo 稳定 value、或改用 useSyncExternalStore/状态库做选择性订阅(只让真正用到那块数据的组件更新)。
本章小结：Context 的值靠一个「值栈」在 Provider 进出时 push/pop;value 变化时 propagateContextChange 从 Provider 向下定向标记依赖该 context 的 fiber,穿透 bailout/memo。→ 下一章:补充·useRef。

C2 · useRef 与 ref 体系

🗺 本章路线图

useRef 是「跨渲染不变的盒子」,改 .current 不触发渲染。

本章包括：useRef 原理、ref 何时被赋值(commit layout)、回调 ref / forwardRef / useImperativeHandle。

useRef:一个「跨渲染不变」的盒子

useRef(init) 在该 Hook 的 memoizedState 上保存一个 { current: init } 对象，并在后续渲染中返回同一个对象。修改 ref.current 不会触发渲染，因为它不属于 state。常见用途包括保存 DOM 节点、定时器 id 和上一次的值。

// mount 时创建盒子;update 时原样返回,引用永不变
function mountRef(initialValue){
  const hook = mountWorkInProgressHook();
  const ref = { current: initialValue };
  hook.memoizedState = ref;
  return ref;
}
// 对比闭包陷阱:ref.current 总是最新的,因为大家共享同一个盒子
useRef vs useState:都能跨渲染保存值。区别:改 state 触发重渲染并参与 UI;改 ref.current 不触发渲染,适合存「不该驱动 UI 的可变值」。

ref 属性:什么时候被赋值

<div ref={myRef}> 里的 myRef.current 不是渲染时赋值的,而是在 Commit 的 Layout 阶段由 React 设置:commitAttachRef 把真实 DOM 赋给 ref.current;卸载或 ref 变化时先 commitDetachRef 置为 null。顺序是先卸旧 ref(=null)、再挂新 ref。所以在 useLayoutEffect 里能拿到 ref,在渲染函数体里拿到的还是上一次的(或 null)。

回调 ref / forwardRef / useImperativeHandle

  • 回调 ref:ref={node => ...},挂载时收到 node、卸载时收到 null。React 19 起回调可返回一个清理函数(替代收到 null)。
  • forwardRef:React 18 及以前,函数组件要接收父级 ref 必须用 forwardRef 包裹。React 19 起 ref 可直接作为 prop,不再需要 forwardRef。
  • useImperativeHandle:配合 ref 只暴露受控的命令式接口(如 { focus, scrollToTop }),而不是把整个 DOM/实例暴露出去,保持封装。
  • string ref(废弃):ref="input" 这种老写法已不推荐,存在性能与语义问题。
// 只暴露 focus,不暴露整个 input 节点
useImperativeHandle(ref, () => ({
  focus(){ inputRef.current.focus(); }
}), []);
本章小结：useRef 存一个 {current} 对象、每次渲染返回同一个(改它不重渲染);ref 在 commit 的 layout 阶段赋值;React 19 起 ref 可直接作 prop。→ 最后:更新流程总览,把整条线串起来。

React 更新流程总览

下面按执行顺序汇总前文涉及的主要步骤。遇到具体问题时，可以从对应步骤返回相关章节：

用户触发更新
  1  生成 Update 对象 + 分配 Lane(据事件类型)          → 更新主线·四个阶段/Schedule·Lane/第五层·合成事件
  2  Update 进 Fiber.updateQueue.shared.pending 环状链表  → Trigger·UpdateQueue
  3  Lane 逐层冒泡,root 收集 pendingLanes                → Schedule·Lane
  4  Scheduler 据 Lane 建任务、入最小堆                    → Schedule·Scheduler
  5  MessageChannel 时间切片,可中断/可抢占               → Schedule·Scheduler/第四层·Concurrent
  6  Render: beginWork 执行 Hooks/算 state/Diff/生成 WIP  → Render·Diff/Render·Hooks/Render·Fiber
  7  completeWork 打 Effect 标记、收集副作用(subtreeFlags)→ Render·Fiber/Render·变更标记
  8  若被高优抢占 → 丢弃 WIP,未执行 Update 迁移 baseQueue → Trigger·UpdateQueue/第四层·Concurrent
